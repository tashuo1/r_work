## 0. 当前代码基线（2026-05-22）

当前仓库已落地一版可运行轻量单体，作为后续迭代基线：

1. `ontology-lite/ontology-server`：`Spring Boot 3 + MyBatis`，产品界面和 API 同服务托管；平台持久化库优先使用 MySQL/PolarDB，外部库不可达时启动脚本自动切换到本地 H2 兜底。
2. `start-ontology-platform.sh`：支持构建、启动、重启、停止、状态查看，默认监听 `127.0.0.1:9000`。
3. 当前已覆盖演示闭环：三库演示数据源、本体建模工作台四子页、扫描失败快照降级提示、候选生成中即时反馈、建模候选、语义图谱、影响评估与页面内发布确认、数据标准、T+1 触发、问数、OpenAPI、验收清单。
4. 当前已支持数据源真实 JDBC 连接信息录入与连接测试；用户需在页面填写财务/投资/人事三套测试源信息。MySQL/PolarDB JDBC URL 会自动补齐连接/读超时参数，前端 API 网络异常显示中文诊断。
5. 当前已新增计算资源管理：支持 DM/MySQL/PolarDB 运行存储目标配置，新增/编辑保存后自动真实 JDBC 测试，连接成功后自动幂等初始化 `ontology_meta`/`ontology_instance`，列表仅保留编辑、设为当前、删除；计算资源连接信息落入 `om_compute_resource / om_compute_resource_health`，active 后平台状态同步写入规范 `om_*` 表。
6. 当前已将“标准词典”升级为“数据标准”：页面包含 `AI生成 / 命名词根 / 标准字段` 三个子页；AI生成支持多选数据源并按 `cizhu-shengcheng` 输出结构生成待审核候选；命名词根和标准字段支持确认/拒绝，`REJECTED` 默认隐藏，确认后同步写入 active 计算资源的标准四表。
7. 当前不作为一期门禁的能力：`ontology-modeling` Skill 实际调用、`cizhu-shengcheng` 真实 LLM 外部执行器、生产级实例 upsert、OAuth2.1/OIDC 鉴权、限流、24h 幂等窗口、Excel 文件导入导出、30分钟后台定时巡检。

## 0. 一期边界（冻结）

1. 仅做一期，不进入二期/三期开发。  
2. 仅接入 3 个源库：农投财务库、农投投资库、农投人事库（达梦）。  
   - 平台连接器标准能力覆盖达梦/MySQL。  
3. 本体实例层仅做 T+1 日更，默认 `01:00`，支持页面手动调整。  
4. 问数执行仅在本体实例层，允许白名单跨库（实例层），禁止跨源库在线 join。  
5. API生产化目标采用成熟方案：OAuth2.1/OIDC Bearer JWT、租户+接口限流、IP兜底、`request_id` 幂等；当前一期先保留请求字段和接口形态，不做治理实现。  
6. P1 用例通过率门槛：`>=90%`。  

## 1. 团队角色与产能假设

| 角色 | 人数 | 主要职责 |
|---|---:|---|
| 产品经理 | 1 | 需求澄清、验收组织、范围控制 |
| 架构师 | 1 | 架构把关、关键方案决策 |
| 后端工程师 | 2 | 连接器、建模服务、ETL、API、安全闸 |
| 前端工程师 | 1 | 建模页面、数据标准页面、问数页面、告警提示 |
| 数据工程师 | 1 | 增量策略、质量门禁、对账回补 |
| 测试工程师 | 1 | 用例执行、回归验证、稳定性验证 |
| 运维/平台 | 0.5 | 调度部署、告警、灰度、回滚演练 |

> 周期：11周（M0~M5），M0与M1部分并行。  
> 估算口径：1人天=1人工作日（不含法定假期）。  
> 框架：一期轻量 Java 单体，模块边界在同一服务内实现。

## 2. WBS编码规则

1. 编码格式：`M{里程碑}-T{任务序号}`，如 `M3-T04`。  
2. 每个任务必须绑定验收用例ID（来自功能清单第7章）。  
3. 每个任务完成需提交：代码/配置变更说明 + 自测记录 + 风险备注。

## 5. 关键依赖链（必须盯住）

1. `M1-T02` -> `M2-T01` -> `M3-T03` -> `M4-T01`（主链路）。  
2. 数据标准治理（M2-T09~T12）会直接影响建模质量与问数准确度。  
3. 安全闸（M4-T02）属于当前一期硬门槛；API治理（M4-T08）为后续生产化目标，当前一期先不做。  

## 6. 每周交付与汇报模板（执行层）

每周固定输出：
1. 本周完成任务（WBS编码）。  
2. 下周计划任务（WBS编码）。  
3. 风险与阻塞（影响里程碑项）。  
4. 验收用例通过率（P0/P1分开统计）。  
5. 是否偏离一期边界（是/否，若是给出纠偏动作）。

## 7. “无脑开发”执行要求（强制）

1. 每个任务必须绑定至少1个验收用例ID。  
2. 任务状态流转：`未开始 -> 开发中 -> 自测通过 -> 联调通过 -> 验收通过`。  
3. 任何代码提交必须附：变更说明、影响范围、回滚说明。  
4. 提测前必须通过对应P0用例；P0不通过不得进入联调。  
5. 不得新增一期范围外能力（如实时CDC、二期审批流、方案B代码生成）。
