### 0.4 安全与治理约束

1. SQL 安全闸保持冻结：仅允许 SELECT、自动 LIMIT、超时控制、执行前预检、禁止跨源库在线 join。
2. 动作触发接口保留 `requestId` 字段；24h 幂等窗口属于后续生产化治理，当前一期先不做。
3. 对外接口当前为本地演示只读接口，生产化阶段再接入正式鉴权、租户、限流和审计网关。
4. 当前轻量版使用 MyBatis + MySQL/PolarDB 持久化平台状态，不连接 Redis、Nacos、消息队列等外部中间件。

### 0.5 当前持久化实现

当前代码已接入 MyBatis，默认通过 `ontology-lite/runtime-config/ontology-platform.env` 使用 MySQL/PolarDB 保存平台运行状态；H2 仅作为无配置时的本地兜底：

| 对象 | 说明 |
|---|---|
| `om_platform_state` | 保存一期演示平台状态 JSON，包括数据源、计算资源、当前生效计算资源ID、词根、标准字段、模型版本、ETL 任务、问数记录、审计日志、T+1 调度 |
| `om_compute_resource` | 保存计算资源中的数据库连接信息；一条计算资源至少落 `META` 与 `INSTANCE` 两条记录，分别对应元数据库/schema 与实例数据库/schema，包含账号、密码、JDBC、初始化状态和生效状态 |
| `om_compute_resource_health` | 保存计算资源数据库连接最近一次测试/初始化健康结果快照，按 `META` / `INSTANCE` 分别记录 |
| `PersistentPlatformStateMapper` | MyBatis 表访问接口 |
| `MyBatisOntologyDataMapper` | 运行时数据访问实现，负责启动加载、状态快照和变更落库 |
| `InMemoryOntologyDataMapper` | 单元测试辅助实现，不作为运行时 Spring Bean |
| `ComputeResourceSchemaInitializer` | 按 DM/MySQL/PolarDB 方言幂等初始化 `ontology_meta` 与 `ontology_instance`，并生成或补齐规范表中文表注释、字段中文注释和枚举值中文说明 |
| `OntologyMetaStore` / `OntologyInstanceStore` | 当前 active 计算资源初始化后，将平台状态同步写入规范 `om_*` 表 |

说明：`om_platform_state` 是轻量版启动兜底与运行状态快照，不等同于 `om_data_source`；`om_data_source` 只保存业务源库规范台账。计算资源连接信息除保留在 JSON 快照中外，同步落入结构化 `om_compute_resource / om_compute_resource_health`；轻量版为支持后续直接使用，`om_compute_resource.password_value` 保存用户填写的连接密码，接口返回和日志仍不展示密码。当用户配置、初始化并激活计算资源后，计算资源中的 `META` / `INSTANCE` 数据库连接信息、数据源、数据标准、模型版本、调度、ETL、问数和审计同步写入 `ontology_meta.om_*`，演示对象实例、关系实例、动作事件同步写入 `ontology_instance.om_*`。初始化幂等，不清空已有库或表数据；MySQL/PolarDB 通过 `CREATE TABLE ... COMMENT` 和 `ALTER TABLE ... COMMENT/MODIFY COLUMN ... COMMENT` 写入或补齐注释，DM 通过 `COMMENT ON TABLE/COLUMN` 写入或补齐注释，枚举字段注释必须同时列出枚举英文值和中文含义。

# 第一部分：接口清单与错误码规范

> 版本：v1（冻结）  
> 关联文档：平台架构设计方案、产品功能清单、WBS

## 1. 通用规范

### 1.1 协议与前缀

1. 协议：当前轻量版本地使用 `HTTP`；生产化部署使用 `HTTPS`。  
2. 内部接口前缀：`/api/v1/internal`。  
3. 对外只读接口前缀：`/api/v1/open`。  
4. 编码：`UTF-8`。  
5. `Content-Type`：`application/json`。

### 1.2 通用请求头

当前轻量版不强制鉴权请求头，前端通过 JSON 字段 `requestId` 传递动作请求标识。生产化目标如下：

1. `Authorization: Bearer <token>`（必填）。  
2. `X-Tenant-Id: <tenant_id>`（必填）。  
3. `X-Request-Id: <request_id>`（动作触发接口必填，其他接口建议传）。  
4. `X-Trace-Id: <trace_id>`（可选，链路追踪）。

### 1.3 通用响应体

当前轻量版响应体：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

生产化目标响应体：

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "request_id": "req_20260520_0001",
  "trace_id": "trace_xxx",
  "timestamp": "2026-05-20T10:00:00+08:00",
  "data": {}
}
```

### 1.4 异步任务响应体

```json
{
  "code": "SUCCESS",
  "message": "accepted",
  "data": {
    "job_id": "job_20260520_0001",
    "status": "PENDING",
    "poll_after_ms": 2000,
    "timeout_seconds": 60
  }
}
```

## 3. 限流、幂等与超时

> 当前一期仅保留 `requestId` 字段和业务审计记录，未强制鉴权、限流、24h 幂等窗口和异步超时；这些能力一期先不做。以下为后续生产化目标规范。

### 3.1 限流（一期默认）

1. 一级限流键：`tenant_id + api_path`。  
2. 二级兜底限流键：`client_ip + api_path`。  
3. 超限返回：`HTTP 429` + `ONT-API-001`。

### 3.2 幂等（动作触发类）

1. 触发类接口必须传 `X-Request-Id`。  
2. 幂等窗口：`24h`。  
3. 重复请求返回首次执行结果摘要，不重复执行任务。  
4. 未传 `request_id` 返回：`HTTP 400` + `ONT-API-002`。

### 3.3 回查超时

1. 异步回查默认超时：`60s`。  
2. 超时返回：`HTTP 200` + `ONT-API-003`（业务超时码，允许继续轮询）。

## 0. 模型边界与分层

1. 本文覆盖 **本体元数据层（meta库）** 和 **本体实例层（instance库）** 完整表结构。  
2. 当前轻量实现通过“计算资源”连接一个数据库实例，并在同一实例内创建两个逻辑库/schema：

## 3.6 数据标准治理域

四张数据标准表的职责边界固定如下：

| 表 | 职责 | 允许存放 | 禁止存放 |
|---|---|---|---|
| `om_term_root` | 词根主数据 | 单一中文业务词的汉语拼音词根 | 标准字段、英文缩写、原始字段名、字段级复合词根 |
| `om_standard_field` | 标准字段主数据 | 一个或多个规范词根组成的字段编码 | 未登记词根、英文 snake_case、源库字段名 |
| `om_root_field_map` | 词根与标准字段的组成/关联关系 | 词根参与标准字段构成的关系 | 词根之间的同义关系 |
| `om_root_synonym` | 词根之间的同义/近义/缩略关系 | 词根到词根的关系 | 标准字段、源库字段、对象名、表名 |

候选词根和候选标准字段不再单独建候选表，通过 `status`、`source_type`、`confidence_level`、`evidence_json`、`review_note` 在四表内治理。`ACTIVE` 表示正式可用，`DRAFT` 表示候选待审核，`REJECTED` 表示已拒绝负样本。

```sql

  root_name           VARCHAR(128) NOT NULL COMMENT '词根中文名(一个原子中文业务词)',

  source_type         VARCHAR(32) DEFAULT NULL COMMENT '来源类型枚举: MANUAL=人工, SCAN=扫描, AI_SKILL=AI技能生成, LLM=模型生成, IMPORT=导入',
  confidence_level    VARCHAR(32) DEFAULT NULL COMMENT '置信等级枚举: L1=高, L2=中, L3=低, CANDIDATE_HIGH=高置信候选, CANDIDATE_LOW=低置信候选, PENDING_REVIEW=待人工复核',

四表强制规则：

1. `om_term_root.root_code` 必须来自中文业务词的汉语拼音，全小写，不允许 `_`、数字、空格、英文单词、英文缩写或源库字段名。
2. `om_term_root` 不能承载字段级语义，例如 `kehumingcheng`、`xiugaishijian`、`danjubianhao`；这类语义必须拆成 `kehu_mingcheng`、`xiugai_shijian`、`danju_bianhao` 标准字段。
3. 辅助识别词根只允许用于 `om_root_synonym.relation_type='ABBREVIATION'`，必须在 `biz_definition` 标明“辅助识别词根/缩略词”，不得参与标准字段编码，也不得作为 `om_root_field_map` 的组成词根。
4. `om_standard_field.field_code` 的每一段都必须能在 `om_term_root.root_code` 中找到，并且只能由规范词根组成。
5. `ACTIVE` 数据默认不可被自动修改；扫描、AI Skill 或 LLM 输出只能先进入 `DRAFT` 或 `REJECTED`，不得绕过审核直接污染正式数据标准。
6. `DRAFT` 标准字段对应的 `om_root_field_map.status` 必须是 `INACTIVE`；`ACTIVE` 标准字段的组成映射应为 `ACTIVE`。
7. `om_root_field_map.relation_type='SYNONYM'` 表示“词根参与字段组成”，不同于 `om_root_synonym.relation_type='SYNONYM'` 的“词根业务等价”，二者不能混用。
8. 覆盖率必须同时报告 `ACTIVE` 覆盖率、`ACTIVE + DRAFT` 候选覆盖率；正式建模只以 `ACTIVE` 为确认口径，候选生成和覆盖审计可读取 `DRAFT` 作为候选上下文。

## 4. 外键与约束建议（一期可选）

> 一期建议先以应用层约束为主，数据库外键为辅，避免跨大表写入阻塞。  
> 若启用外键，建议优先在元数据静态表启用。

建议外键关系（元数据层）：
1. `om_object_type.model_version_id -> om_model_version.id`
2. `om_property.object_type_id -> om_object_type.id`
3. `om_link_type.source_object_id/target_object_id -> om_object_type.id`
4. `om_link_property.link_type_id -> om_link_type.id`
5. `om_action_param.action_type_id -> om_action_type.id`
6. `om_mapping_rule.object_type_id/property_id -> om_object_type/om_property.id`
7. `om_root_synonym.root_id/synonym_root_id -> om_term_root.id`
8. `om_object_identifier.object_type_id -> om_object_type.id`

实例层（instance 库）不启用数据库外键，应用层保证引用完整性。

## 5. 分区与索引建议

1. 元数据层：`om_etl_run_log`、`om_change_log`、`om_quality_result`、`om_query_history` 建议按月分区。
2. 实例层：`om_object_instance`、`om_relation_instance`、`om_action_event` 建议按 `biz_date` / `event_time` 月分区。
3. 高频查询索引：
   - `tenant_id + model_version_id + status`
   - `tenant_id + api_name`
   - `tenant_id + job_id`
   - `tenant_id + object_type_id + is_deleted`（实例查询）
4. 审计表保留策略：在线 90 天，历史归档。

## 8. 落地约束（已确认，v2 更新）

1. 当前轻量实现通过一个 active 计算资源承载 meta 与 instance：MySQL/PolarDB 创建 `ontology_meta`、`ontology_instance` 两个 database；DM 创建同名 schema。  
2. `om_platform_state` 保留为启动兜底与运行状态快照；active 计算资源初始化后，同步写入规范 `om_*` 表；旧 `ontology_platform_state` 已废弃删除，不再自动创建或兼容读取。  
3. 元数据中心库一期优先支持 MySQL/PolarDB；DM 计算资源使用兼容 schema 初始化。  
4. `JSON` 字段在目标 MySQL 规范中采用原生 `JSON`；当前轻量实现为兼容 DM/H2，DDL 初始化使用 `LONGTEXT`/`CLOB` 存储 JSON 内容。  
5. 数据库外键 meta 库可选开启，instance 库关闭（应用层保障一致性）。  
6. 多租户一期采用逻辑隔离（`tenant_id`）。  
7. 审计表按”在线90天+归档”策略执行。
8. 单对象属性数上限 50，超过需冷热分离评估（高频字段独立列，低频字段留 JSON）。
9. 软删除检测：T+1 增量覆盖 + 每周日全量 COUNT(*) 对账，差异标记 `is_deleted=1` 并推送人工确认。
10. 虚拟列 DDL 执行窗口：T+1 批跑完成后到下一批次启动前。

## 9. 模块级工程落地规范（Codex 终审，v2 新增）

> 本节将每个产品模块的数据库操作精确到表、字段、状态值、操作顺序，研发可直接按此编写 Service/DAO 代码。
> 约定：`[meta]` = 元数据库，`[instance]` = 实例库。所有操作均带 `tenant_id`，下文中省略。

### 9.1 数据源管理模块

#### 9.1.1 新增数据源（POST /api/v1/internal/datasources）

```
操作顺序：
0. 页面从“新增数据源”弹层选择 DM 后进入“创建DM数据源”表单，填写基础信息、生产环境连接参数和 JDBC 连接串预览；驱动类、默认参数、高级参数由后台展示或按类型推导，前台不展示；认证文件管理不参与一期流程。数据源列表行仅展示“编辑/删除”，MySQL/PolarDB JDBC 查询参数在列表中隐藏，完整连接串仍用于后台连接测试；MySQL/PolarDB 默认生成和手工填写的 JDBC URL 均补齐 `connectTimeout=10000` 与 `socketTimeout=10000`。
1. INSERT INTO [meta].om_data_source
   source_code  = 用户输入（如 'FINANCE_DB'）
   source_name  = 用户输入（如 '农投财务库'）
   db_type      = 'DM'
   status       = 'ACTIVE'
   其他连接参数 = 用户输入
   → 返回 datasource_id

2. 不写 om_data_source_health（首次巡检时写入）
3. 不写 om_change_log（数据源管理操作暂不纳入本体模型变更审计，如需要可后续扩展 entity_type=DATASOURCE）
```

#### 9.1.2 连接测试（POST /api/v1/internal/datasources/{id}/test）

```
1. 读取 [meta].om_data_source WHERE id = {id} → 获取连接参数
2. 使用连接参数尝试建立 JDBC 连接，记录耗时 ms；MySQL/PolarDB 连接串必须带 `connectTimeout=10000` 与 `socketTimeout=10000`，避免数据库网络不可达时页面长时间等待。
3. INSERT INTO [meta].om_data_source_health
   source_id     = {id}
   check_time    = NOW()
   health_status = 连接成功 ? 'OK' : 具体错误（'TIMEOUT'/'CONNECTION_REFUSED'/'AUTH_FAILED'）
   error_code    = 失败时的错误码（如 'ONT-DS-003'）
   error_message = 失败时的异常信息
   response_ms   = 连接耗时
4. UPDATE [meta].om_data_source SET last_health_time = NOW() WHERE id = {id}
```

#### 9.1.3 定时巡检（每 30 分钟）

```
1. SELECT id, host, port, db_name, schema_name, username, secret_ref
   FROM [meta].om_data_source WHERE status = 'ACTIVE'
2. 对每个数据源执行连接测试（同 9.1.2 步骤 2-4）
3. 如果 health_status != 'OK'，前端显示红色感叹号（数据源管理页面读取 last_health_time 和最新 health_status）
```

#### 9.1.4 停用数据源（POST /api/v1/internal/datasources/{id}/toggle）

```
1. UPDATE [meta].om_data_source SET status = 'INACTIVE' WHERE id = {id}
2. 停用后，巡检任务跳过此数据源（health_check 查询条件 status='ACTIVE'）
```

#### 9.1.5 删除数据源（DELETE /api/v1/internal/datasources/{id}）

```
1. 读取 [meta].om_data_source WHERE id = {id}，不存在则返回 ONT-DS-002。
2. 删除数据源配置，并同步移除该数据源已保存的密码/密钥引用。
3. 写入 DATASOURCE/DELETE 审计记录，返回被删除的数据源摘要。
```

#### 9.2.2 候选模型生成（POST /api/v1/internal/modeling/candidates/generate）

这是整个平台最核心的写入流程，涉及 6 张表：

```
1. 创建新语义版本
   INSERT INTO [meta].om_model_version
   version_no  = 'v_{timestamp}'（如 v_20260520_140000）
   status      = 'INIT'
   is_latest   = 1
   → 返回 model_version_id

2. 将上一个 is_latest=1 的版本置为 is_latest=0（保留历史）
   UPDATE [meta].om_model_version SET is_latest = 0
   WHERE tenant_id = {tenant_id} AND is_latest = 1 AND id != {model_version_id}

3. 记录 ETL 日志
   INSERT INTO [meta].om_etl_run_log
   job_id            = 'modeling_{timestamp}_{uuid}'
   job_type          = 'MODELING'
   source_id         = {数据源ID}
   model_version_id  = {model_version_id}
   status            = 'RUNNING'
   start_time        = NOW()

4. 对扫描结果中的每张表，生成对象候选
   INSERT INTO [meta].om_object_type
   model_version_id   = {model_version_id}
   object_code        = 自动生成（如 'FIN_VOUCHER'，规则：{DOMAIN}_{TABLE_ABBR}）
   api_name           = 自动生成（如 'FinanceVoucher'）
   display_name       = LLM 生成（如 '会计凭证'）
   domain_name        = 自动识别（finance/investment/hr/cross_domain）
   status             = 'DRAFT'
   primary_key_policy = 'BUSINESS_KEY'
   confidence_score   = rule_score × 0.6 + llm_score × 0.4（见 5.1.1 节公式）
   source_ref_json    = '{"schema":"FINANCE","table":"VOUCHER","source_id":1}'
   → 返回 object_type_id

5. 对每张表的每个字段，生成属性候选
   INSERT INTO [meta].om_property
   model_version_id   = {model_version_id}
   object_type_id     = {对应对象ID}
   property_code      = 自动生成（如 'voucher_no'）
   api_name           = 自动生成（如 'voucherNo'）
   display_name       = LLM 生成（如 '凭证号'）
   data_type          = JDBC 类型映射（VARCHAR→string, NUMBER→decimal, DATE→date...）
   is_required        = 源库 NOT NULL 约束 ? 1 : 0
   is_searchable      = 默认 1
   enum_values_json   = NULL（Schema-Only 建模不填枚举值，人工可补充）
   source_ref_json    = '{"schema":"FINANCE","table":"VOUCHER","column":"VOUCHER_NO"}'
   confidence_score   = rule_score × 0.6 + llm_score × 0.4
   status             = 'DRAFT'

6. 检测源库外键约束，生成关系候选
   INSERT INTO [meta].om_link_type
   model_version_id    = {model_version_id}
   link_code           = 自动生成（如 'VOUCHER_TO_ENTRY'）
   api_name            = 自动生成（如 'VoucherHasEntry'）
   display_name        = LLM 生成（如 '凭证包含明细'）
   source_object_id    = {外键父表对应的 object_type_id}
   target_object_id    = {外键子表对应的 object_type_id}
   cardinality         = '1:N'（外键默认 1:N）
   link_category       = 'FOREIGN_KEY'
   confidence_score    = 9.5（外键是高置信度）
   source_ref_json     = '{"fk_name":"FK_VOUCHER_ENTRY","source_table":"VOUCHER","target_table":"VOUCHER_ENTRY"}'
   status              = 'DRAFT'

7. 生成对象标识规则
   INSERT INTO [meta].om_object_identifier
   model_version_id   = {model_version_id}
   object_type_id     = {对象ID}
   identifier_name    = 'business_key'
   property_codes     = 主键列对应的 property_code（逗号分隔）
   is_primary         = 1
   status             = 'ACTIVE'

8. 生成属性映射规则（每个属性一条）
   INSERT INTO [meta].om_mapping_rule
   model_version_id    = {model_version_id}
   source_id           = {数据源ID}
   object_type_id      = {对象ID}
   property_id         = {属性ID}
   source_schema       = 源库 schema
   source_table        = 源表
   source_column       = 源字段
   increment_strategy  = 自动判断：有 update_time → 'A'，有 biz_date → 'B'，否则 → 'C'
   mapping_status      = 'PENDING'
   confidence_score    = {属性置信度}
   status              = 'DRAFT'

9. 记录建模完成
   UPDATE [meta].om_etl_run_log
   SET status = 'SUCCESS', end_time = NOW(), duration_ms = {耗时}
   WHERE job_id = {job_id}
```

**自动通过阈值判定**（不满足阈值的标记为 DRAFT 而不是 ACTIVE）：

| 场景 | 自动通过阈值 | 写入 status |
|---|---|---|
| 属性类型映射 | confidence >= 9.5 | DRAFT（需人工确认才变 ACTIVE） |
| 对象识别 | confidence >= 8.5 | DRAFT |
| 关系推断（外键） | confidence >= 8.5 | DRAFT |
| 业务含义推断 | confidence >= 7.5 | DRAFT |
| Action/Event 推断 | confidence >= 7.0 | DRAFT |

注意：所有候选模型初始状态均为 `DRAFT`，**不自动生效**。必须通过 9.2.3 步骤人工确认后才变为 `ACTIVE`。

#### 9.2.3 人工建模确认（前端多轮修正 + 生效）

```
场景 A：直接确认（无需修改）
  1. 前端展示候选对象/属性/关系，用户浏览后点击"确认生效"
  2. → 进入 9.4 发布流程

场景 B：修正后确认（多轮对话）
  1. 用户在前端发起修正指令（重命名/合并/删除/重映射）
  2. POST /api/v1/internal/modeling/candidates/{id}/revise
  3. 后端：
     UPDATE [meta].om_object_type SET
       display_name = '新名称', updated_at = NOW() WHERE id = {id}
     （或其他修正操作）
  4. INSERT INTO [meta].om_change_log
     change_type  = 'UPDATE'
     entity_type  = 'OBJECT'（或 PROPERTY/LINK）
     entity_id    = {实体ID}
     entity_code  = {实体编码}
     before_json  = '{"display_name":"旧名称"}'
     after_json   = '{"display_name":"新名称"}'
     reason_code  = 'RENAME'
     operator_id  = {当前用户}
     op_time      = NOW()
  5. 修正后用户确认 → 进入 9.4 发布流程

场景 C：驳回（需填写驳回原因）
  1. 用户选择"驳回"
  2. INSERT INTO [meta].om_change_log
     change_type  = 'REJECT'
     entity_type  = 'OBJECT'（或 PROPERTY/LINK）
     entity_id    = {实体ID}
     reason_code  = {驳回原因分类，必填}
     operator_id  = {当前用户}
     op_time      = NOW()
     before_json  = {当前候选内容快照}
  3. 驳回的对象/属性/关系保持 status = 'DRAFT'，下次建模重新评估
```

### 9.3 数据标准治理模块

数据标准治理不单独创建候选表。候选命名词根和候选标准字段通过四表内状态治理：

| 字段/状态 | 含义 |
|---|---|
| `status='ACTIVE'` | 已确认正式可用 |
| `status='DRAFT'` | 候选，待人工审核 |
| `status='REJECTED'` | 已拒绝负样本，后续生成器不得反复生成 |
| `source_type` | `MANUAL=人工` / `SCAN=扫描` / `AI_SKILL=AI技能生成` / `LLM=模型生成` / `IMPORT=导入` |
| `confidence_level` | `L1=高` / `L2=中` / `L3=低` / `CANDIDATE_HIGH=高置信候选` / `CANDIDATE_LOW=低置信候选` / `PENDING_REVIEW=待人工复核` |
| `evidence_json` | 字段名、注释、表上下文、样例值、分词结果、枚举等证据 |
| `review_note` | 审核说明、拒绝原因、下一步动作 |

#### 9.3.1 新增词根

```
INSERT INTO [meta].om_term_root
root_code      = 用户输入（校验：必须全小写拼音、不包含下划线、不包含数字）
root_name      = 用户输入
biz_definition = 用户输入
domain_name    = 用户选择
source_type    = 'MANUAL' / 'SCAN' / 'AI_SKILL' / 'LLM' / 'IMPORT'
confidence_level = 'L1' / 'L2' / 'L3' / 'CANDIDATE_HIGH' / 'CANDIDATE_LOW' / 'PENDING_REVIEW'
evidence_json  = {证据JSON，可为空}
review_note    = {审核说明}
status         = 'ACTIVE'（人工确认）或 'DRAFT'（候选待审核）或 'REJECTED'（负样本）

INSERT INTO [meta].om_change_log
change_type  = 'CREATE'
entity_type  = 'TERM_ROOT'
entity_code  = {root_code}
after_json   = {完整词根内容JSON}
operator_id  = {当前用户}
op_time      = NOW()
```

#### 9.3.2 新增标准字段

```
INSERT INTO [meta].om_standard_field
field_code     = 用户输入（支持小写拼音+下划线组合，如 'yonghu_mingcheng'）
field_name     = 用户输入
field_type     = 用户选择
biz_definition = 用户输入
combo_key      = 系统自动生成（格式：{自增编号}_{field_code}，如 '00001_yonghu_mingcheng'）
source_type    = 'MANUAL' / 'SCAN' / 'AI_SKILL' / 'LLM' / 'IMPORT'
confidence_level = 'L1' / 'L2' / 'L3' / 'CANDIDATE_HIGH' / 'CANDIDATE_LOW' / 'PENDING_REVIEW'
evidence_json  = {证据JSON，可为空}
review_note    = {审核说明}
status         = 'ACTIVE'（正式字段）或 'DRAFT'（候选字段）或 'REJECTED'（负样本）

INSERT INTO [meta].om_change_log（entity_type = 'STANDARD_FIELD'）
```

#### 9.3.3 词根与标准字段映射

```
INSERT INTO [meta].om_root_field_map
root_id          = {词根ID，用户选择}
standard_field_id = {标准字段ID，用户选择}
relation_type    = 'SYNONYM'（默认，表示词根参与标准字段组成）
weight_score     = 1.00（默认）
status           = 标准字段为 ACTIVE 时取 'ACTIVE'；标准字段为 DRAFT 时必须取 'INACTIVE'
```

#### 9.3.4 词根同义词管理

```
INSERT INTO [meta].om_root_synonym
root_id         = {主词根ID（如 shouji）}
synonym_root_id = {同义词根ID（如 dianhua）}
relation_type   = 'SYNONYM' / 'NEAR' / 'ABBREVIATION'
weight_score    = 用户可选（默认 1.00）
status          = 'ACTIVE'

-- 查询一个词根的所有同义词：
SELECT r.root_code, r.root_name, s.relation_type, s.weight_score
FROM om_root_synonym s
JOIN om_term_root r ON s.synonym_root_id = r.id
WHERE s.root_id = {词根ID} AND s.status = 'ACTIVE'
```

#### 9.3.5 数据标准导入（后续 Excel/CSV）

```
1. 解析上传文件，逐行校验
2. 人工确认导入：批量 INSERT INTO om_term_root / om_standard_field（status='ACTIVE', source_type='MANUAL'）
3. 扫描/AI Skill/LLM 候选导入：批量 INSERT INTO om_term_root / om_standard_field（status='DRAFT', source_type='SCAN'/'AI_SKILL'/'LLM'）
4. 校验失败或明确拒绝的：可写入 status='REJECTED' 作为负样本，也可生成错误明细返回下载链接
5. INSERT INTO [meta].om_change_log（change_type='CREATE', entity_type='TERM_ROOT'/'STANDARD_FIELD'）
```

**数据标准生效时机**：只有 `status='ACTIVE'` 的词根和标准字段视为正式生效。`DRAFT` 可用于候选覆盖率、人工审核和下一轮 Skill 学习，但不作为最终确认口径。建模 Skill 在下次建模任务（POST /api/v1/internal/modeling/candidates/generate）时读取 ACTIVE 作为强上下文，读取 DRAFT 作为候选上下文。已生效的语义模型不受数据标准更新影响。

#### 9.3.6 工程落库与质量校验

1. 批量导入或程序生成时，必须按 `om_term_root` -> `om_standard_field` -> `om_root_field_map` -> `om_root_synonym` 的顺序入库。
2. 入库前必须校验 `root_code` 是否为单一汉语拼音词，`field_code` 是否能完整拆回已登记规范词根，是否混入英文单词、英文缩写、数字、空格或特殊符号。
3. `om_root_field_map` 的 `SYNONYM` 映射必须覆盖标准字段编码的全部词根段。
4. `om_root_synonym` 禁止自关联，禁止同一关系重复双向写入；`ABBREVIATION` 只能连接主词根与辅助识别词根。
5. 程序读取数据标准用于正式建模时，只能读取 `status='ACTIVE'`；用于候选生成和覆盖审计时，可以读取 `DRAFT` 作为候选上下文。
6. 无中文注释、无已确认缩写、无可靠上下文时，系统必须输出“待治理”或 `PENDING_REVIEW`，不得假装自动生成成功。
7. 平台元数据、配置数据、模板数据、流程数据、技术数据默认不得进入正式业务标准字段，除非对应表被人工确认为业务对象。

### 9.4 语义确认与发布模块

#### 9.4.1 生效前影响评估

```
1. 计算影响范围
   SELECT COUNT(*) AS affected_objects
   FROM [meta].om_object_type
   WHERE model_version_id = {当前版本ID} AND status = 'DRAFT'

   SELECT COUNT(*) AS affected_links
   FROM [meta].om_link_type
   WHERE model_version_id = {当前版本ID} AND status = 'DRAFT'

   SELECT COUNT(*) AS affected_tasks
   FROM [meta].om_mapping_rule mr
   JOIN [meta].om_object_type ot ON mr.object_type_id = ot.id
   WHERE mr.model_version_id = {当前版本ID} AND ot.status = 'DRAFT'

2. 写回版本表
   UPDATE [meta].om_model_version SET
   impact_summary_json = JSON_OBJECT(
     'affected_objects', {对象数},
     'affected_links', {关系数},
     'affected_tasks', {任务数}
   )
   WHERE id = {model_version_id}

3. 前端展示影响评估（三个数字），用户二次确认
```

#### 9.4.2 生效发布（单人确认，二次确认后触发）

```
1. 版本状态变更
   UPDATE [meta].om_model_version
   SET status = 'ACTIVE', effective_at = NOW(), effective_by = {操作人}
   WHERE id = {model_version_id}

2. 批量更新该版本下所有 DRAFT 实体为 ACTIVE
   UPDATE [meta].om_object_type SET status = 'ACTIVE'
   WHERE model_version_id = {model_version_id} AND status = 'DRAFT'
   
   UPDATE [meta].om_property SET status = 'ACTIVE'
   WHERE model_version_id = {model_version_id} AND status = 'DRAFT'
   
   UPDATE [meta].om_link_type SET status = 'ACTIVE'
   WHERE model_version_id = {model_version_id} AND status = 'DRAFT'

   UPDATE [meta].om_link_property SET status = 'ACTIVE'
   WHERE model_version_id = {model_version_id} AND status = 'DRAFT'
   
   UPDATE [meta].om_mapping_rule SET status = 'ACTIVE'
   WHERE model_version_id = {model_version_id} AND status = 'DRAFT'

3. 将该版本的所有历史版本 status 改为 INVALID
   UPDATE [meta].om_model_version SET status = 'INVALID'
   WHERE tenant_id = {tenant_id} AND status = 'ACTIVE' AND id != {model_version_id}

4. 审计日志
   INSERT INTO [meta].om_change_log
   change_type  = 'PUBLISH'
   entity_type  = 'VERSION'
   entity_id    = {model_version_id}
   entity_code  = {version_no}
   before_json  = {上一 ACTIVE 版本快照}
   after_json   = {当前版本快照}
   operator_id  = {操作人}
   op_time      = NOW()

5. ETL日志记录
   INSERT INTO [meta].om_etl_run_log
   job_id     = 'publish_{timestamp}_{uuid}'
   job_type   = 'PUBLISH'
   model_version_id = {model_version_id}
   status     = 'SUCCESS'
   start_time = NOW()
   end_time   = NOW()
```

#### 9.4.3 回滚（POST /api/v1/internal/modeling/rollback-latest）

```
1. 校验：仅允许回滚最近 ACTIVE 版本，单次不超过 10 个对象/条目
   SELECT id FROM [meta].om_model_version
   WHERE tenant_id = {tenant_id} AND status = 'ACTIVE'
   ORDER BY effective_at DESC LIMIT 1
   → 返回最近生效版本 ID

   SELECT COUNT(*) FROM [meta].om_object_type
   WHERE model_version_id = {最近生效版本ID}
   → 校验 <= 10

2. 将最近生效版本标记为 INVALID
   UPDATE [meta].om_model_version SET status = 'INVALID'
   WHERE id = {最近生效版本ID}

3. 找到前一个 ACTIVE 版本，重新激活
   SELECT id FROM [meta].om_model_version
   WHERE tenant_id = {tenant_id} AND status = 'INVALID'
   ORDER BY effective_at DESC LIMIT 1
   
   UPDATE [meta].om_model_version
   SET status = 'ACTIVE', is_latest = 1, rollback_from_id = {被回滚的版本ID}
   WHERE id = {前一个版本ID}

4. 将前一个版本的实体重新设为 ACTIVE
   UPDATE [meta].om_object_type SET status = 'ACTIVE'
   WHERE model_version_id = {前一个版本ID}
   -- 同理处理 om_property, om_link_type, om_link_property, om_mapping_rule

5. INSERT INTO [meta].om_change_log（change_type='ROLLBACK', ...）
```

### 9.5 T+1 实例更新模块

#### 9.5.1 T+1 调度触发（每日 01:00）

```
1. 内置调度服务触发任务入口
   job_id = 't1_{biz_date}_{timestamp}'
   biz_date = DATE_SUB(CURRENT_DATE, 1)  -- D-1 日数据

2. INSERT INTO [meta].om_etl_run_log
   job_id     = {job_id}
   job_type   = 'ETL'
   biz_date   = {biz_date}
   status     = 'PENDING'
   start_time = NOW()

3. 读取生效语义版本
   SELECT id AS model_version_id, version_no
   FROM [meta].om_model_version
   WHERE tenant_id = {tenant_id} AND status = 'ACTIVE' AND is_latest = 1
   → model_version_id 快照，整个 T+1 过程使用此快照

4. UPDATE [meta].om_etl_run_log
   SET status = 'RUNNING', model_version_id = {model_version_id}

5. 加载 DAG：从语义层读取生效的对象和关系
   SELECT * FROM [meta].om_object_type
   WHERE model_version_id = {model_version_id} AND status = 'ACTIVE'
   
   SELECT * FROM [meta].om_link_type
   WHERE model_version_id = {model_version_id} AND status = 'ACTIVE'
   
   SELECT * FROM [meta].om_mapping_rule
   WHERE model_version_id = {model_version_id} AND status = 'ACTIVE'

6. 按依赖顺序生成 DAG（对象任务 → 关系任务 → 质量门禁 → 发布快照）
```

#### 9.5.2 增量抽取 + 转换 + 装载（核心链路）

```
对 DAG 中的每个对象类型，按 mapping_rule 的增量策略执行：

【策略 A：水位增量】
  1. SELECT watermark_value, biz_date
     FROM [meta].om_etl_watermark
     WHERE source_id = {source_id}
       AND source_schema = {schema}
       AND source_table = {table}
       AND strategy_type = 'A'

  2. 在源库执行增量抽取 SQL（只读）：
     SELECT * FROM {schema}.{table}
     WHERE update_time > '{watermark_value}'
     ORDER BY update_time ASC

  3. 对每一行源数据：
     a) 构建 properties_json = {"api_name_1": row.get("COL1"), "api_name_2": row.get("COL2"), ...}
        （对照 om_mapping_rule 中的 source_column → api_name 映射）
     b) instance_id = 拼接 business_key：{company_code}_{voucher_no}（对照 om_object_identifier）
     c) 查前一版本的 properties_json（用于对比，生成 action_event）：
        SELECT properties_json FROM [instance].om_object_instance
        WHERE object_type_id = {object_type_id} AND instance_id = {instance_id}
        AND is_deleted = 0
        → 存储为 before_json
     d) 执行 upsert：
        INSERT INTO [instance].om_object_instance
        (object_type_id, object_api_name, instance_id, display_label,
         properties_json, batch_id, model_version_id, biz_date)
        VALUES (...)
        ON DUPLICATE KEY UPDATE
        properties_json = VALUES(properties_json),
        batch_id = VALUES(batch_id),
        model_version_id = VALUES(model_version_id),
        is_deleted = 0,
        updated_at = NOW()
     e) 如果 before_json 和 properties_json 有差异：
        INSERT INTO [instance].om_action_event
        event_id        = 'evt_{timestamp}_{uuid}'
        object_type_id  = {object_type_id}
        object_instance_id = {instance_id}
        action_type_id  = NULL（STATUS_CHANGE 关联的动作类型，若无则为 NULL）
        event_type      = 'STATUS_CHANGE'
        event_time      = NOW()
        before_json     = {变更前属性值}
        after_json      = {变更后属性值}
        changed_fields_json = {变更字段列表}
        batch_id        = {batch_id}

  4. 更新水位
     UPDATE [meta].om_etl_watermark
     SET watermark_value = '{本次抽取到的最大 update_time}',
         biz_date = {biz_date},
         last_success_time = NOW()
     WHERE source_id = {source_id}
       AND source_schema = {schema}
       AND source_table = {table}

【策略 B：业务日期分区】
  同策略 A，但增量条件改为：
  WHERE biz_date = '{biz_date}'

【策略 C：小表全量】
  同策略 A，但增量条件改为：
  全量抽取（无 WHERE 条件），或按主键范围分片抽取

【多线程并发】
  - 每库初始并发 4 线程
  - 同库内不同表分片并行
  - 关系任务的 DAG 依赖确保：对象 upsert 完成后才执行关系 upsert
```

#### 9.5.3 关系实例构建

```
1. 对每个生效关系类型 (om_link_type.status='ACTIVE')：

   a) 外键来源 (link_category='FOREIGN_KEY')：
      SELECT source.instance_id AS src_id, target.instance_id AS tgt_id
      FROM [instance].om_object_instance source
      JOIN [instance].om_object_instance target
        ON source.properties_json->>'$.{source_fk_column}' = target.properties_json->>'$.{target_pk_column}'
      WHERE source.object_type_id = {source_object_type_id}
        AND target.object_type_id = {target_object_type_id}
        AND source.batch_id = {batch_id}

   b) 将 JOIN 结果写入关系实例表：
      INSERT INTO [instance].om_relation_instance
      (link_type_id, link_api_name, source_object_type_id, source_instance_id,
       target_object_type_id, target_instance_id, properties_json,
       start_date, batch_id, model_version_id)
      VALUES (...)
      ON DUPLICATE KEY UPDATE
      properties_json = VALUES(properties_json),
      batch_id = VALUES(batch_id),
      updated_at = NOW()
```

#### 9.5.4 质量门禁

```
对所有对象类型的 upsert 结果，执行质量门禁：

1. SELECT
   COUNT(*) AS row_count,
   COUNT(DISTINCT instance_id) AS unique_keys
   FROM [instance].om_object_instance
   WHERE object_type_id = {object_type_id} AND batch_id = {batch_id}

2. 计算指标：
   - row_delta：|本批行数 - 上批行数| / 上批行数
   - dup_rate：(总行数 - 唯一键数) / 总行数
   - null_rate：从 properties_json 计算空值字段数 / 总字段数（仅监控）

3. INSERT INTO [meta].om_quality_result
   job_id         = {job_id}
   object_type_id = {object_type_id}
   metric_code    = 'row_delta' / 'dup_rate' / 'null_rate'
   metric_value   = {计算值}
   threshold_value = {阈值：row_delta=0.20, dup_rate=0}
   pass_flag      = 实际值 <= 阈值 ? 1 : 0
   severity       = 超阈值 ? 'HIGH' : 'INFO'
   detail_json    = {详细行数/重复键列表}

4. 门禁判断：
   - 所有对象 row_delta 通过 AND dup_rate 通过 → 门禁通过
   - 任一对象 dup_rate > 0 → 阻断发布，告警（severity='CRITICAL'）
   - 任一对象 row_delta > 0.20 → 阻断发布，告警（severity='HIGH'）
   - null_rate 仅记录，不阻断

5. 通过门禁：
   INSERT INTO [meta].om_etl_run_log
   job_id  = 'quality_{batch_id}'
   job_type = 'QUALITY'
   status  = 'SUCCESS' / 'FAILED'
   
   然后 INSERT INTO [meta].om_etl_run_log（job_type='SNAPSHOT', status='SUCCESS'）
   标记本批次为"已发布快照"
```

#### 9.5.5 软删除检测（周度）

```
每 7 天（周日 02:00，T+1 完成后）执行全量对账：

1. 对每个数据源-对象类型组合：
   SELECT COUNT(*) FROM {源库}.{源表} → 源库行数
   SELECT COUNT(*) FROM [instance].om_object_instance WHERE object_type_id = {id} AND is_deleted = 0 → 实例库行数

2. 如果源库行数 < 实例库行数：存在物理删除
   SELECT instance_id FROM [instance].om_object_instance
   WHERE object_type_id = {id} AND is_deleted = 0
   → 与源库所有主键比对，找到缺失的 instance_id

3. UPDATE [instance].om_object_instance SET is_deleted = 1, updated_at = NOW()
   WHERE object_type_id = {id} AND instance_id IN ({缺失ID列表})

4. INSERT INTO [meta].om_quality_result
   metric_code  = 'deleted_rows'
   metric_value = {缺失行数}
   severity     = 'WARNING'
   detail_json  = {缺失ID列表JSON}
```

#### 9.6.1 问数提交（POST /api/v1/internal/query/jobs）

```
1. 创建问数记录

   query_id         = 'qry_{timestamp}_{uuid}'
   request_id       = {X-Request-Id}
   raw_question     = {用户自然语言问题}

   → 异步返回 job_id

   SELECT ot.display_name, ot.api_name, p.display_name, p.api_name, p.data_type
   FROM [meta].om_object_type ot
   JOIN [meta].om_property p ON ot.id = p.object_type_id

     AND p.status = 'ACTIVE'

3. LLM 生成 SQL → generated_sql
   UPDATE [meta].om_query_history
   SET generated_sql = {生成的SQL}, status = 'RUNNING'

4. SQL 安全闸校验
   a) 仅允许 SELECT → 否则拦截，status='SECURITY_BLOCKED'
   b) 检查目标表是否在实例层白名单 schema 内
   c) 检查是否跨源业务库 join（禁止）
   d) 自动注入 LIMIT 1000 + 超时 30s
   e) EXPLAIN 预检

   SET executed_sql = {最终执行SQL},
       execution_ms = {耗时},
       result_rows  = {行数},
