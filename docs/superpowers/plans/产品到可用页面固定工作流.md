# 产品到可用页面固定工作流

## 目标

用于所有“产品功能、页面、工具、后台管理界面、SaaS 界面、交互应用”的需求。

最终结果必须满足：

- 有明确产品目标和目标用户。
- 有可执行 PRD、用户故事、验收标准和边界场景。
- 页面不是静态摆设，主流程可用。
- 页面符合真实用户使用习惯，信息层级清楚，操作路径短。
- 视觉上克制、专业、有美感，不依赖空洞装饰。
- 完成后必须做桌面和移动端视觉检查。

## 固定执行顺序

### 1. 产品判断

先回答：

- 用户是谁？
- 用户要完成什么任务？
- 当前替代方案是什么？
- 为什么现在的方案值得做？
- 成功指标是什么？
- 哪些不做？

推荐使用：

```text
product-human-workflow
office-hours
define-problem-statement
define-hypothesis
foundation-persona
discover-journey-map
```

产物：

- 问题定义
- 目标用户
- JTBD
- 成功指标
- 非目标

门禁：

```text
目标用户和主流程不清楚，不进入设计和开发。
```

## 2. 产品文档

推荐使用：

```text
deliver-prd
deliver-user-stories
deliver-acceptance-criteria
deliver-edge-cases
develop-solution-brief
```

必须产出：

- PRD
- 用户故事
- 验收标准
- 边界场景
- 空状态、加载状态、错误状态、权限失败状态
- 数据字段、权限、审计要求

门禁：

```text
验收标准必须可测试，不能只写“体验好”“页面美观”。
```

## 3. 范围审查

推荐使用：

```text
plan-ceo-review
utility-pm-critic
```

默认策略：

- 企业后台、运营工具：保持范围，优先可用、稳定、信息密度。
- 面向用户的产品：允许选择性扩展，增强体验闭环。

门禁：

```text
范围不能大到无法完成，也不能小到主流程不可用。
```

## 4. 设计系统与交互

推荐使用：

```text
design-consultation
plan-design-review
develop-design-rationale
```

必须定义：

- 信息架构
- 页面布局
- 主操作路径
- 组件规范
- 字体、颜色、间距、密度
- 空、加载、错误、成功状态
- 桌面和移动端适配策略

硬性规则：

- 默认做真实可用页面，不做营销落地页。
- 后台和 SaaS 页面要克制、密集、可扫描。
- 不嵌套卡片。
- 不用装饰性大色块、渐变球、无意义插画撑页面。
- 表格、筛选、详情、弹窗、批量操作必须符合常见工作流。
- 文字不能溢出、重叠或遮挡。
- 主按钮、次按钮、危险按钮必须明确区分。
- 页面第一屏必须能看出用户下一步该做什么。

门禁：

```text
用户 5 秒内看不懂下一步怎么操作，设计不合格。
```

## 5. 实现

推荐使用：

```text
design-html
```

开发要求：

- 先读现有代码和组件。
- 复用现有设计系统。
- 主流程必须可点、可查、可提交或可完成。
- 不能只交静态视觉稿。
- 有开发服务器的项目必须启动并给出访问地址。

门禁：

```text
没有主流程交互，不算完成。
```

## 6. 审查与验收

推荐使用：

```text
design-review
qa
utility-pm-critic
```

必须验证：

- 桌面端。
- 移动端。
- 主流程。
- 空状态。
- 加载状态。
- 错误状态。
- 权限或禁用状态。
- 文字不重叠、不溢出。
- 操作按钮可识别。
- 验收标准逐条通过。

门禁：

```text
没有视觉验证，不能声称“美观”。
没有主流程验证，不能声称“可用”。
```

## 推荐调用方式

当用户提出：

- 做一个产品功能
- 设计一个页面
- 写 PRD
- 做后台管理页
- 做 SaaS 页面
- 做一个工具
- 要符合人类使用
- 要有美感

优先使用：

```text
product-human-workflow
```

再按阶段调用：

```text
pm-skills -> gstack 产品审查 -> gstack 设计 -> 实现 -> 设计审查 -> QA
```

## 最终交付格式

如果是产品文档任务：

```text
1. 产品简报
2. PRD
3. 用户故事
4. 验收标准
5. 边界场景
6. 设计方向
7. 实施清单
8. 验收清单
```

如果是开发实现任务：

```text
1. 改了什么
2. 改在哪里
3. 如何验证
4. 当前访问地址
5. 剩余风险
```
