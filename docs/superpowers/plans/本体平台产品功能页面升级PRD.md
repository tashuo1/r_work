# 本体平台产品功能页面升级 PRD

> 版本：v1  
> 日期：2026-05-23  
> 范围：基于当前 `ontology-lite/ontology-server` 产品页面，升级为对标 DataWorks 工作流与 Palantir Foundry Ontology 的可用本体平台界面。  
> 工作流依据：`docs/superpowers/plans/产品到可用页面固定工作流.md`

## 0. 当前审查结论

当前产品已经具备一期演示闭环：工作台、数据源、计算资源、本体建模、数据标准、T+1 同步、语义问数、OpenAPI、验收清单均可访问，并且多数按钮连接真实 API，不是纯静态页面。

但当前形态仍偏“轻量演示台”，尚未达到“可对标 DataWorks 的生产工作台”和“可对标 Palantir Foundry 的 Ontology 操作系统”：

1. DataWorks 对标不足：缺 DataStudio / Operation Center / Data Map / Data Quality / DataService Studio 级别的信息架构和任务生命周期。
2. Palantir 对标不足：已有 Object / Property / Link 雏形，但缺 Object View、Action Type、Function、Role、Action Log 与对象实例探索。
3. 产品信任问题：启动脚本在外部平台库不可达时会切换到 H2 兜底，但页面没有强提示，用户容易误判“数据被删除”。
4. 视觉质感问题：当前界面干净但仍偏后台模板，距离“像苹果一样美观、克制、清晰、有质感”还有差距。
5. 人体使用问题：当前页面能点，但用户无法始终在 5 秒内知道“下一步该做什么、失败后怎么恢复、当前状态是否可信”。

## 1. 产品判断

### 1.1 目标用户

| 用户 | 核心任务 | 关键页面 |
|---|---|---|
| 数据架构师 | 建立本体对象、关系、标准字段、版本和发布规则 | Ontology Studio、标准建模、版本发布 |
| 数据工程师 | 接入源库、配置同步、排查质量失败和运行任务 | 数据集成、管理中心、运维中心 |
| 业务分析师 | 基于业务对象问数、查看口径、导出结果 | 分析工作台、对象视图 |
| 治理管理员 | 管理权限、审计、API、验收和发布风险 | 服务发布、发布验收、审计 |

### 1.2 用户要完成的主任务

一期产品主流程必须被页面清晰承载：

```text
接入源库 -> 初始化平台计算资源 -> 扫描元数据 -> 生成/审核数据标准 -> 生成本体模型 -> 影响评估并发布 -> T+1 同步 -> 语义问数 -> API 服务发布 -> 验收
```

### 1.3 当前替代方案

1. DataWorks / Data Integration：解决数据接入、开发、调度、治理，但不直接提供企业本体对象模型。
2. 传统数据中台：用表、字段、指标管理业务数据，业务语义和对象关系较弱。
3. Palantir Foundry：以 Ontology 组织企业业务对象和动作，但本项目需要本地化、轻量化、面向现有三库快速验证。

### 1.4 为什么值得做

当前页面已经证明后端闭环可运行，下一阶段最大风险不是“能不能做接口”，而是“产品是不是像一个真实平台”。如果页面不能像 DataWorks 一样组织数据开发工作流、不能像 Palantir 一样组织对象语义，业务用户和数据开发人员仍会把它当成演示系统，而不是可长期工作的本体平台。

### 1.5 成功指标

| 指标 | 目标 |
|---|---|
| 首屏可信度 | 用户进入工作台 5 秒内能看清当前环境、持久化库、异常和下一步任务 |
| 主流程完成度 | 9 个主模块串成一条可操作闭环，不出现“只有入口没有流程”的页面 |
| DataWorks 对标度 | 数据集成、运维中心、数据质量、服务发布具备任务生命周期和运行态视图 |
| Palantir 对标度 | Ontology Studio 支持对象、属性、关系、动作、实例、版本和审计视角 |
| 视觉质量 | 桌面和移动端无文字重叠、无无意义装饰、主次危险操作明确 |
| 验收质量 | P0 用例 100% 通过；页面每个主流程有空、加载、错误、成功状态 |

### 1.6 非目标

1. 一期不做实时 CDC。
2. 一期不实现生产级 OAuth2.1/OIDC、租户限流和 24h 幂等窗口。
3. 一期不扩展 3 个源库之外的大规模生产接入。
4. 一期不做复杂审批流，保持单人确认生效。
5. 不做营销落地页，不用装饰性视觉替代真实工作流。

## 2. 解决方案简报

将当前“演示工作台”升级为“本体数据操作系统”：

1. 用 DataWorks 式的工作台结构承载数据接入、开发、调度、质量、服务发布。
2. 用 Palantir 式的 Ontology Studio 承载对象、属性、关系、动作、实例、版本和审计。
3. 用 Apple 式的视觉原则重塑界面：低噪音、高精度、克制颜色、清晰层级、明确反馈。
4. 用“当前环境可信状态条 + 今日待办 + 右侧 Inspector + 工作流状态机”提升人体使用效率。

### 2.1 核心产品结构

| 当前页面 | 升级后页面名 | 对标 | 核心变化 |
|---|---|---|---|
| 工作台 | 运营总览 | DataWorks 首页 + Foundry 首页 | 从展示型 Hero 改为今日待办和可信状态控制台 |
| 数据源 | 数据集成 | DataWorks Data Integration | 新增源库接入向导、Schema 预览、健康趋势、连接测试日志 |
| 计算资源 | 管理中心 | DataWorks Management Center | 展示 active 资源、H2/外部库状态、初始化和规范表同步 |
| 本体建模 | Ontology Studio | Palantir Ontology | 新增对象详情、实例视图、Action、Function、Role 占位 |
| 数据标准 | 标准建模 | DataWorks Data Modeling | 候选队列、冲突处理、影响对象、负样本沉淀 |
| T+1 同步 | 运维中心 | DataWorks Operation Center | DAG、任务实例、质量门禁、日志、重跑、watermark |
| 语义问数 | 分析工作台 | DataAnalysis + Foundry Query | 对象集查询、口径解释、SQL 安全闸解释、追问历史 |
| OpenAPI | 服务发布 | DataService Studio | API 目录、测试台、发布状态、调用示例、错误码 |
| 验收清单 | 发布验收 | 发布门禁 | P0/P1、截图证据、接口证据、环境、风险 |

## 3. PRD

### 3.1 运营总览

目标：用户进入系统后第一眼知道当前平台是否可信、哪些任务需要处理、下一步该做什么。

功能要求：

1. 顶部固定环境状态条：
   - 当前平台持久化库：MySQL/PolarDB/H2。
   - 当前 JDBC 主机或 H2 文件路径。
   - 是否处于兜底模式。
   - 最近一次切换原因。
   - 当前 active 计算资源。
2. 今日待办：
   - 待配置数据源。
   - 待初始化计算资源。
   - 待审核数据标准。
   - 待发布本体版本。
   - 失败 T+1 任务。
   - API 未发布或验收失败项。
3. 平台健康：
   - 数据源健康。
   - 计算资源健康。
   - 元数据库/实例库初始化状态。
   - 最近 ETL 状态。
   - 当前本体版本。
4. 主流程进度条：
   - 接入源库。
   - 计算资源。
   - 标准建模。
   - 本体建模。
   - T+1。
   - 问数。
   - API。
   - 验收。

验收标准：

- 给定外部库不可达并切换 H2，用户在首页顶部能看到“当前使用 H2 兜底库”和原因。
- 给定存在 PENDING 验收项，首页待办区必须显示对应处理入口。
- 首页不得以大面积营销式 Hero 占据主工作区。

### 3.2 数据集成

目标：对标 DataWorks 数据源管理，完成从源库连接到可用于建模/同步的接入流程。

功能要求：

1. 数据源列表：
   - 数据源名称、类型、业务域、连接地址、状态、最近测试、Schema 数、表数、负责人、操作。
   - 演示数据必须标记 `DEMO`，真实数据源标记 `REAL`。
2. 新增数据源向导：
   - 选择数据源类型。
   - 填写连接信息。
   - 执行网络连通性测试。
   - 执行只读权限测试。
   - 展示 Schema 预览。
   - 保存并发布。
3. 健康详情：
   - 最近连接测试结果。
   - 失败原因。
   - 恢复建议。
   - 最近使用它的建模/T+1 任务。
4. 错误状态：
   - 网络失败。
   - DNS 解析失败。
   - 账号密码错误。
   - 权限不足。
   - 驱动缺失。

验收标准：

- 新增数据源必须至少经过“连接测试”才能保存为可用状态。
- 连接失败时必须保留输入值并展示中文恢复建议。
- 移动端数据源列表必须转为记录卡片，不依赖横向表格阅读。

### 3.3 管理中心

目标：让用户知道平台运行存储在哪里、是否初始化完成、当前写入目标是否可靠。

功能要求：

1. 当前资源卡：
   - active 资源名称。
   - 元数据库。
   - 实例库。
   - 初始化表数量。
   - 最近健康状态。
   - 最近写入时间。
2. 计算资源列表：
   - DM/MySQL/PolarDB 类型。
   - 连接地址。
   - 初始化状态。
   - active 状态。
   - 最近巡检。
3. 初始化状态机：
   - 待测试。
   - 测试中。
   - 测试失败。
   - 初始化中。
   - 初始化成功。
   - 部分成功，可重试。
4. H2 兜底说明：
   - H2 只能作为演示兜底。
   - 外部平台库恢复后如何切回。

验收标准：

- 当系统使用 H2 时，管理中心必须明确显示“非生产持久化”。
- active 计算资源禁止删除，删除按钮需禁用并说明原因。

### 3.4 Ontology Studio

目标：对标 Palantir Foundry Ontology，把本体从“模型候选列表”升级为“业务对象操作系统”。

功能要求：

1. 左侧资产树：
   - Object Types。
   - Link Types。
   - Action Types。
   - Functions。
   - Roles。
   - Versions。
2. 中间主工作区：
   - 图谱视图。
   - 表格视图。
   - 版本差异视图。
   - 对象实例视图。
3. 右侧 Inspector：
   - 对象基本信息。
   - 属性映射。
   - 源表血缘。
   - 标准字段。
   - 实例数量。
   - 质量规则。
   - API 暴露状态。
   - 审计日志。
4. 发布流程：
   - 影响对象数。
   - 影响关系数。
   - 影响 T+1 任务。
   - 影响 API。
   - 二次确认。
5. Action Type 一期口径：
   - 一期先只做只读定义和审计展示。
   - 不执行写源库动作。

验收标准：

- 任意对象类型都可以打开详情 Inspector。
- 任意对象详情能看到源表、标准字段和版本状态。
- 发布前必须展示影响评估，不允许无提示直接生效。

### 3.5 标准建模

目标：把当前 AI 生成、命名词根、标准字段升级为可审核、可解释、可复用的数据标准工作台。

功能要求：

1. AI 候选队列：
   - 来源数据源。
   - 候选词根。
   - 候选标准字段。
   - 置信度。
   - 证据。
   - 冲突原因。
   - 相似词根。
2. 审核工作流：
   - 确认。
   - 拒绝。
   - 合并到已有词根。
   - 标记为辅助识别词根。
3. 影响分析：
   - 确认后影响哪些对象。
   - 下次建模是否会引用。
   - 是否同步 active 计算资源标准四表。
4. 负样本沉淀：
   - 被拒绝候选保留原因。
   - 后续生成时降低同类候选优先级。

验收标准：

- 每个候选都必须有证据和审核动作。
- 拒绝动作必须要求选择或填写原因。
- 标准字段必须能拆回已登记词根。

### 3.6 运维中心

目标：对标 DataWorks Operation Center，让 T+1 同步从“结果表”升级为“可排障、可重跑、可追踪”的任务中心。

功能要求：

1. DAG 视图：
   - 对象抽取。
   - 关系构建。
   - 质量门禁。
   - 快照发布。
2. 任务实例列表：
   - 实例 ID。
   - 状态。
   - 开始/结束时间。
   - 耗时。
   - 处理行数。
   - 失败原因。
3. 节点详情：
   - 日志。
   - watermark。
   - 输入/输出。
   - 关联数据源。
   - 关联对象。
4. 操作：
   - 手动触发。
   - 重跑失败节点。
   - 查看质量报告。
   - 修改调度。

验收标准：

- 任意失败任务能定位到失败节点和中文失败原因。
- 质量门禁失败时不得显示为成功。
- watermark 必须和数据源、对象任务关联展示。

### 3.7 分析工作台

目标：让业务用户通过本体对象问数，并理解结果口径。

功能要求：

1. 问数输入：
   - 自然语言问题。
   - 对象范围选择。
   - 时间范围。
   - 常用问题模板。
2. 结果解释：
   - 摘要。
   - 明细。
   - 口径。
   - 使用对象。
   - 使用本体版本。
   - 生成 SQL。
   - 安全闸说明。
3. 多轮追问：
   - 保留上下文。
   - 显示继承条件。
4. 导出：
   - CSV。
   - SQL。
   - 查询解释。

验收标准：

- 每次问数结果必须展示使用的对象类型和本体版本。
- SQL 安全闸拦截时必须说明触发规则。
- 图表切换后不丢失原始明细和 SQL。

### 3.8 服务发布

目标：把 OpenAPI 从结果展示页升级为 API 发布和测试中心。

功能要求：

1. API 目录：
   - 元数据 API。
   - 对象实例 API。
   - 问数结果 API。
   - 动作触发 API。
2. API 详情：
   - 请求路径。
   - 请求参数。
   - 响应字段。
   - 错误码。
   - 调用示例。
   - 鉴权状态。
   - 发布状态。
3. 测试控制台：
   - 输入参数。
   - 发送请求。
   - 展示响应。

验收标准：

- 本地无鉴权必须明确标注为演示环境能力。
- 每个 API 至少提供一个可执行示例。

### 3.9 发布验收

目标：把验收清单升级为发布门禁。

功能要求：

1. P0/P1 统计。
2. 失败项聚合。
3. 每个用例的证据：
   - API 证据。
   - 页面截图。
   - 操作时间。
   - 剩余风险。
4. 环境状态：
   - 当前持久化库。
   - 当前版本。
   - 当前 active 计算资源。

验收标准：

- P0 未全部通过时，页面必须显示“不可发布”。
- 每个失败项必须能跳转到对应模块。

## 4. 用户故事

### US-01 数据工程师接入源库

作为数据工程师，我想通过向导录入源库连接并完成权限校验，以便后续建模和 T+1 同步基于可信数据源运行。

验收：

- Given 我在数据集成页点击新增数据源，When 填写连接信息并测试成功，Then 系统展示 Schema 预览并允许保存。
- Given 连接测试失败，When 我查看错误结果，Then 页面展示中文原因和恢复建议，并保留我的输入。

### US-02 架构师发布本体版本

作为数据架构师，我想在发布前看到对象、关系、任务和 API 的影响范围，以便确认不会破坏现有业务链路。

验收：

- Given 存在候选版本，When 我点击确认生效，Then 系统必须展示影响评估。
- Given 我取消发布，Then 模型版本不变化，审计记录不写入生效动作。

### US-03 业务分析师问数

作为业务分析师，我想用自然语言查询业务对象，并看到结果口径和安全闸说明，以便信任查询结果。

验收：

- Given 我提交问题，When 查询成功，Then 页面展示摘要、明细、SQL、对象类型、本体版本和口径说明。
- Given SQL 被安全闸拦截，Then 页面展示拦截规则和可恢复建议。

### US-04 管理员判断环境可信

作为治理管理员，我想一眼看出当前系统使用的是外部 MySQL/PolarDB 还是 H2 兜底库，以便判断页面数据是否可信。

验收：

- Given 外部库不可达并切换 H2，When 我进入工作台，Then 顶部环境条显示 H2 兜底和切换原因。
- Given 外部库恢复，When 服务连接外部库，Then 顶部环境条显示外部库主机和库名。

## 5. 边界与错误状态

| 场景 | 页面行为 |
|---|---|
| 外部平台库不可达 | 工作台和管理中心显示 H2 兜底警示，不把演示数据伪装成真实数据 |
| 数据源 DNS 失败 | 数据源详情显示 DNS 解析失败和检查建议 |
| 数据源账号密码错误 | 连接测试失败，保留输入，不保存为可用状态 |
| 数据源权限不足 | 只读校验失败，提示缺少 SELECT / 元数据读取权限 |
| 计算资源初始化部分成功 | 显示部分成功表清单，允许重试，不允许设为 active |
| AI 候选生成失败 | 标准建模页展示失败状态和重试入口 |
| 发布影响评估失败 | 禁止发布，展示失败原因 |
| T+1 质量门禁失败 | DAG 节点标红，整体任务状态不能显示 SUCCESS |
| 问数 SQL 被拦截 | 展示触发规则，不执行 SQL |
| 移动端查看表格 | 表格转卡片或折叠详情，不出现横向阅读负担 |

## 6. 设计方向

### 6.1 总体原则

1. 像 Apple 一样克制：高留白但不空洞，层级精确，颜色只服务状态和动作。
2. 像 DataWorks 一样高效：筛选、表格、详情、任务、日志、运行态都为频繁操作设计。
3. 像 Palantir 一样对象化：用户围绕对象、关系、动作、实例工作，而不是只围绕表和接口工作。
4. 不做营销页，不使用装饰性大色块、渐变球和无意义插画。

### 6.2 页面布局标准

所有主工作页采用统一结构：

```text
顶部环境状态条
页面标题 + 主动作
筛选/模式工具条
主工作区
右侧 Inspector / 详情抽屉
底部或侧边运行反馈
```

### 6.3 视觉系统

| 项 | 规则 |
|---|---|
| 背景 | `#f5f7fa` 到 `#ffffff`，避免大面积彩色渐变 |
| 主色 | 深蓝/系统蓝，仅用于主动作和当前选中 |
| 成功 | 绿色，仅用于健康或通过 |
| 警告 | 黄色/琥珀，仅用于待处理或风险 |
| 危险 | 红色，仅用于删除、阻断、失败 |
| 卡片 | 仅用于重复项、详情、弹窗，不嵌套卡片 |
| 表格 | 桌面高密度，移动端转卡片 |
| 图标 | 使用统一线性图标，替换临时字符按钮 |

## 9. 范围审查结论

采用“选择性扩展”：

1. 保持一期边界，不进入实时 CDC、生产鉴权、复杂审批流。
2. 必须扩展产品页面深度，否则无法达成 DataWorks / Palantir 对标。
3. 第一阶段优先修产品信任、环境状态、工作台待办和本体对象详情。
4. 暂不追求全量 Foundry 能力，但必须在页面模型中出现 Object View、Action、Function、Role 的结构位置。

## 10. 当前风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| H2 兜底和外部库切换不透明 | 用户误判数据丢失 | 首页和管理中心强提示 |
| 页面深度继续停留在演示 | 业务方不信任平台 | 以 DataWorks / Palantir 信息架构重构主页面 |
| Apple 式美观被误解为低密度营销页 | 降低后台效率 | 坚持高密度、低噪音、任务型界面 |
| Palantir 能力过度扩张 | 一期失控 | Action/Function/Role 先做只读结构和占位，不做执行 |
| 移动端表格可读性差 | 巡检场景不可用 | 表格转卡片，筛选折叠 |
