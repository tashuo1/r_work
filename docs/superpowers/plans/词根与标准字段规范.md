# 词根与标准字段规范

> 适用范围：本体论数据平台一期词典治理、语义建模、字段归一与问数口径统一。  
> 核心原则：**词根是中文业务词的汉语拼音；标准字段由词根组合而成。**

## 1. 基本定义

### 1.1 词根

词根是最小可复用语义单元，对应一个中文业务词。

词根编码使用该中文词的**汉语拼音**，全小写。

本规范把词根分为两类：

| 类型 | 定义 | 是否可组成标准字段 |
|---|---|---|
| 规范词根 | 中文业务词的完整汉语拼音 | 可以 |
| 辅助识别词根 | 经评审保留的拼音首字母、历史简称、系统惯用缩写 | 不可以，只能用于 `om_root_synonym` 辅助识别 |

除明确标注为“辅助识别词根”的场景外，本文所说“词根”默认指**规范词根**。

示例：

| 中文业务词 | 词根编码 |
|---|---|
| 用户 | `yonghu` |
| 名称 | `mingcheng` |
| 编码 | `bianma` |
| 编号 | `bianhao` |
| 金额 | `jine` |
| 日期 | `riqi` |
| 时间 | `shijian` |
| 项目 | `xiangmu` |
| 合同 | `hetong` |
| 组织 | `zuzhi` |

### 1.2 标准字段

标准字段是完整字段语义，由一个或多个词根组成。

多个词根之间使用英文下划线 `_` 连接。

示例：

| 中文字段含义 | 拆解词根 | 标准字段编码 |
|---|---|---|
| 用户名称 | `yonghu` + `mingcheng` | `yonghu_mingcheng` |
| 客户编码 | `kehu` + `bianma` | `kehu_bianma` |
| 手机号码 | `shouji` + `haoma` | `shouji_haoma` |
| 合同编号 | `hetong` + `bianhao` | `hetong_bianhao` |
| 创建时间 | `chuangjian` + `shijian` | `chuangjian_shijian` |

## 1.3 四张表的职责边界

| 表 | 职责 | 允许存放 | 禁止存放 |
|---|---|---|---|
| `om_term_root` | 词根主数据 | 单一中文业务词的汉语拼音词根 | 标准字段、英文缩写、原始字段名 |
| `om_standard_field` | 标准字段主数据 | 一个或多个词根组成的字段编码 | 未登记词根、英文 snake_case、源库字段名 |
| `om_root_field_map` | 词根与标准字段的组成/关联关系 | 词根参与标准字段构成的关系 | 词根之间的同义关系 |
| `om_root_synonym` | 词根之间的同义/近义/缩略关系 | 词根到词根的关系 | 标准字段、源库字段、对象名、表名 |

强制边界：

1. `om_term_root` 管“词”。
2. `om_standard_field` 管“字段”。
3. `om_root_field_map` 管“词如何组成字段”。
4. `om_root_synonym` 管“词与词之间如何互相替代或辅助识别”。

## 1.3.1 四表内候选治理

本项目不单独创建候选表。词根候选、标准字段候选、分词器候选与 LLM 候选，均通过四表自身的状态和证据字段承载。

`om_term_root` 与 `om_standard_field` 应增加并维护以下治理字段：

| 字段 | 含义 |
|---|---|
| `source_type` | 来源：`RULE` / `JIEBA` / `LLM` / `MANUAL` |
| `confidence_level` | 置信等级：`AUTO` / `L1` / `L2` / `L3` / `CANDIDATE_HIGH` / `CANDIDATE_LOW` / `PENDING_REVIEW` |
| `evidence_json` | 字段名、注释、表上下文、样例值、分词结果、枚举值等证据 |
| `review_note` | 审核说明、拒绝原因、下一步动作 |

状态含义：

| 状态 | 含义 | 是否可用于正式业务 |
|---|---|---|
| `ACTIVE` | 已确认正式词根/标准字段 | 是 |
| `DRAFT` | 候选词根/候选标准字段，待人工审核 | 可用于候选覆盖和审核，不视为最终确认 |
| `REJECTED` | 已明确拒绝的负样本 | 否 |
| `INACTIVE` | 映射或关系未生效 | 否 |

强制规则：

1. `ACTIVE` 数据默认不可被自动修改。
2. `DRAFT` 标准字段对应的 `om_root_field_map.status` 必须是 `INACTIVE`。
3. `REJECTED` 数据保留为负样本，后续生成器不得反复生成。
4. jieba、AliNLP、LLM 的输出只能先进入 `DRAFT` 或 `REJECTED`，不得直接进入 `ACTIVE`。
5. 覆盖率需同时报告 `ACTIVE` 覆盖率、`ACTIVE + DRAFT` 候选覆盖率。

## 1.3.2 当前四表结构与使用方式

### `om_term_root`：词根表

用于保存原子中文业务词的拼音编码，是标准字段的组成基础。

当前字段：

| 字段 | 含义 | 使用说明 |
|---|---|---|
| `id` | 主键 | 自增 |
| `tenant_id` | 租户ID | 多租户隔离 |
| `root_code` | 词根编码 | 汉语拼音、小写、不可包含 `_` |
| `root_name` | 词根中文名 | 一个原子中文业务词 |
| `biz_definition` | 业务定义 | 说明词根含义、来源和口径 |
| `domain_name` | 业务域 | 如 `common`、`finance`、`hr`、`investment` |
| `source_type` | 来源 | `RULE` / `JIEBA` / `LLM` / `MANUAL` |
| `confidence_level` | 置信等级 | `AUTO` / `L1` / `L2` / `L3` / `PENDING_REVIEW` 等 |
| `evidence_json` | 证据JSON | 分词、样例字段、来源表、枚举等证据 |
| `review_note` | 审核说明 | 人工审核意见、拒绝原因、下一步动作 |
| `status` | 状态 | `ACTIVE` / `DRAFT` / `REJECTED` |
| `created_at` | 创建时间 | 系统生成 |
| `updated_at` | 更新时间 | 系统更新 |

使用规则：

1. `ACTIVE` 词根可组成正式标准字段。
2. `DRAFT` 词根只表示候选词根，可用于候选覆盖和人工审核。
3. `REJECTED` 词根是负样本，生成器不得反复生成。
4. 词根不能表达字段级语义，例如 `kehumingcheng`、`xiugaishijian` 不允许入表。

### `om_standard_field`：标准字段表

用于保存由词根组合成的标准字段编码。

当前字段：

| 字段 | 含义 | 使用说明 |
|---|---|---|
| `id` | 主键 | 自增 |
| `tenant_id` | 租户ID | 多租户隔离 |
| `field_code` | 标准字段编码 | 一个或多个词根用 `_` 连接 |
| `field_name` | 标准字段中文名 | 词根中文名组合后的字段含义 |
| `field_type` | 字段类型 | 如 `STRING`、`NUMBER`、`DATE` |
| `biz_definition` | 口径说明 | 字段含义、来源示例、覆盖字段数、枚举值 |
| `combo_key` | 排序/组合键 | 用于稳定排序或导入批次标识 |
| `source_type` | 来源 | `RULE` / `JIEBA` / `LLM` / `MANUAL` |
| `confidence_level` | 置信等级 | `AUTO` / `L1` / `L2` / `L3` / `PENDING_REVIEW` 等 |
| `evidence_json` | 证据JSON | 来源字段、分词结果、样例值、枚举分布 |
| `review_note` | 审核说明 | 待确认原因、驳回原因、审核动作 |
| `status` | 状态 | `ACTIVE` / `DRAFT` / `REJECTED` |
| `created_at` | 创建时间 | 系统生成 |
| `updated_at` | 更新时间 | 系统更新 |

使用规则：

1. `field_code` 的每一段都必须能在 `om_term_root.root_code` 中找到。
2. `ACTIVE` 标准字段是正式可用字段。
3. `DRAFT` 标准字段是候选字段，可用于覆盖分析和人工审核。
4. jieba/LLM/规则生成但未确认的字段只能进入 `DRAFT`。
5. 报表指标型字段默认进入 `DRAFT`，人工确认后才能升为 `ACTIVE`。

### `om_root_field_map`：词根-标准字段映射表

用于保存“哪些词根组成了哪个标准字段”。

当前字段：

| 字段 | 含义 | 使用说明 |
|---|---|---|
| `id` | 主键 | 自增 |
| `tenant_id` | 租户ID | 多租户隔离 |
| `root_id` | 词根ID | 指向 `om_term_root.id` |
| `standard_field_id` | 标准字段ID | 指向 `om_standard_field.id` |
| `relation_type` | 关系类型 | 标准字段组成关系默认使用 `SYNONYM` |
| `weight_score` | 权重 | 默认 `1.00` |
| `status` | 状态 | `ACTIVE` / `INACTIVE` |
| `created_at` | 创建时间 | 系统生成 |
| `updated_at` | 更新时间 | 系统更新 |

使用规则：

1. `ACTIVE` 标准字段的组成映射应为 `ACTIVE`。
2. `DRAFT` 标准字段的组成映射必须为 `INACTIVE`。
3. 这里的 `SYNONYM` 表示“词根参与字段组成”，不是词根之间的同义关系。
4. 词根之间的同义、近义、缩略关系只能放在 `om_root_synonym`。

### `om_root_synonym`：词根同义关系表

用于保存词根与词根之间的同义、近义、缩略关系。

当前字段：

| 字段 | 含义 | 使用说明 |
|---|---|---|
| `id` | 主键 | 自增 |
| `tenant_id` | 租户ID | 多租户隔离 |
| `root_id` | 主词根ID | 指向 `om_term_root.id` |
| `synonym_root_id` | 关联词根ID | 指向另一个 `om_term_root.id` |
| `relation_type` | 关系类型 | `SYNONYM` / `NEAR` / `ABBREVIATION` |
| `weight_score` | 相似权重 | 默认 `1.00` |
| `status` | 状态 | `ACTIVE` / `INACTIVE` |
| `created_at` | 创建时间 | 系统生成 |
| `updated_at` | 更新时间 | 系统更新 |

使用规则：

1. 只能保存词根到词根的关系。
2. 不得保存标准字段、源字段、表名、对象名。
3. `NEAR` 表示近义，不等于完全可替换。
4. `ABBREVIATION` 表示缩略或历史简称，只能辅助识别，不能直接参与标准字段编码。

## 1.4 词根合法不等于标准字段必须入库

一个中文词可以是合法词根，但由它组成的字段不一定应进入正式标准字段。

判断标准字段是否入库，必须同时满足：

1. 词根合法。
2. 字段是业务对象的业务属性。
3. 字段不是平台元数据、配置数据、模板数据、流程数据、技术数据。
4. 字段能在当前业务域中形成可复用口径。

示例：

| 中文字段 | 词根拆解 | 标准字段 | 是否入库 | 原因 |
|---|---|---|---|---|
| 表标题 | `biao` + `biaoti` | `biao_biaoti` | 默认不入库 | 通常是平台元数据/报表模板字段 |
| 表类型 | `biao` + `leixing` | `biao_leixing` | 默认不入库 | 通常是配置/元数据字段 |
| 客户类型 | `kehu` + `leixing` | `kehu_leixing` | 可入库 | 客户业务属性 |
| 项目类型 | `xiangmu` + `leixing` | `xiangmu_leixing` | 可入库 | 项目业务属性 |

因此：

```text
表 是合法词根；
表标题/表类型 也能形成标准字段编码；
但它们默认不属于业务标准字段，除非该表被人工确认为业务对象。
```

## 2. 词根编码规范

词根字段：`om_term_root.root_code`

必须满足：

1. 必须来自中文业务词。
2. 必须使用汉语拼音。
3. 必须全小写。
4. 必须是单一语义单元。
5. 不允许下划线。
6. 不允许数字、空格、特殊符号。
7. 不允许英文单词、英文缩写、数据库字段缩写直接作为词根。
8. 汉语拼音不加声调，不使用数字声调。
9. 多音字按当前中文业务词的实际读音取拼音。
10. 同一个 `tenant_id` 下，一个 `root_code` 只能代表一个中文业务词。

### 2.0 词根禁止承载字段级语义

词根不能表达完整字段口径。凡是中文含义明显由“主体词 + 属性词”构成的，不得作为一个词根写入 `om_term_root`。

字段级语义的典型后缀包括：

```text
时间、日期、名称、编号、编码、类型、状态、金额、号码、代码、标志、标识、标题、说明、地址
```

如果中文名以这些属性词结尾，且前面还有主体词，必须拆分：

| 错误词根 | 中文含义 | 正确拆分 | 正确标准字段 |
|---|---|---|---|
| `xiugaishijian` | 修改时间 | `xiugai` + `shijian` | `xiugai_shijian` |
| `caozuoshijian` | 操作时间 | `caozuo` + `shijian` | `caozuo_shijian` |
| `danjubianhao` | 单据编号 | `danju` + `bianhao` | `danju_bianhao` |
| `kehumingcheng` | 客户名称 | `kehu` + `mingcheng` | `kehu_mingcheng` |
| `jiaoyijine` | 交易金额 | `jiaoyi` + `jine` | `jiaoyi_jine` |

允许作为单一词根的情况：

1. 该中文词本身是稳定业务术语，如“身份证”“营业执照”。
2. 该中文词不是“主体词 + 属性词”的字段结构。
3. 该词已作为权威业务术语被确认，并有业务定义。

禁止事项：

1. 不允许因为标准字段需要映射，就自动把完整字段中文名写入词根表。
2. 不允许 `om_term_root` 中出现大量以 `shijian`、`riqi`、`mingcheng`、`bianhao`、`bianma`、`leixing`、`zhuangtai`、`jine`、`haoma`、`biaoshi`、`biaoti` 结尾的复合词根。
3. 如果出现上述复合词根，必须视为词根污染，进入清洗或重生成流程。

例外规则：

1. `om_term_root` 可以存放少量**辅助识别词根**，用于承接 `om_root_synonym.relation_type = 'ABBREVIATION'`。
2. 辅助识别词根必须来自中文词的拼音首字母或组织内已固化简称，例如 `sfz` 对应“身份证”。
3. 辅助识别词根不得参与标准字段编码。
4. 辅助识别词根不得在 `om_root_field_map` 中以 `SYNONYM` 关系作为标准字段组成词根。
5. 辅助识别词根必须在 `biz_definition` 中标明“辅助识别词根/缩略词”，并由人工确认。

正确示例：

```text
yonghu
mingcheng
kehu
bianma
bianhao
riqi
shijian
jine
zhuangtai
xiangmu
hetong
```

错误示例：

```text
user
name
user_name
phone_num
id_card
cust
createTime
USER_NAME
yonghu_mingcheng
```

说明：

- `yonghu_mingcheng` 不是词根，它是标准字段。
- `user_name` 不是标准字段，因为它由英文组成，不是汉语拼音词根组合。
- 若同一拼音可能对应多个中文词，必须优先选择业务上更明确的中文词作为词根，必要时走人工评审，不允许让一个 `root_code` 同时代表两个中文含义。

### 2.1 词根选择标准

新增词根时按以下优先级选择：

1. 优先选择业务人员自然使用的中文名词。
2. 优先选择能跨表、跨库复用的稳定语义。
3. 优先选择单一概念，不把修饰词和中心词合成一个词根。
4. 对组织内已有标准术语，优先沿用已有中文术语。
5. 对缩写、系统字段、英文字段，必须先还原中文业务含义，再生成汉语拼音词根。

示例：

| 源字段/表达 | 中文业务含义 | 词根选择 |
|---|---|---|
| `USER_NAME` | 用户、名称 | `yonghu`、`mingcheng` |
| `CUST_NO` | 客户、编号 | `kehu`、`bianhao` |
| `ORG_CODE` | 组织、编码 | `zuzhi`、`bianma` |
| `AMT` | 金额 | `jine` |

不得这样选择：

| 错误选择 | 问题 |
|---|---|
| `user` | 英文，不是汉语拼音 |
| `cust` | 英文缩写，不是汉语拼音 |
| `yonghumingcheng` | 合并了两个词根，不可拆解 |
| `yonghu_mingcheng` | 这是标准字段，不是词根 |

### 2.2 两层词根体系

词根体系采用“两层建设”：

```text
基础通用词根层 + 业务域词根层
```

先建设基础通用词根，再从业务数据库中补充业务域词根。

这样做的目的：

1. 提高中文分词质量。
2. 减少字段级复合词根污染。
3. 提高标准字段收敛度。
4. 提高物理字段映射覆盖率。
5. 降低业务词根人工确认成本。

#### 2.2.1 基础通用词根层

基础通用词根是跨业务域、跨系统、高复用的词根，应在扫描业务库前初始化。

建议首批包括以下类别：

| 类别 | 中文词 | 词根编码 |
|---|---|---|
| 日期时间类 | 日期 | `riqi` |
| 日期时间类 | 时间 | `shijian` |
| 日期时间类 | 年 | `nian` |
| 日期时间类 | 月 | `yue` |
| 日期时间类 | 日 | `ri` |
| 日期时间类 | 开始 | `kaishi` |
| 日期时间类 | 结束 | `jieshu` |
| 日期时间类 | 创建 | `chuangjian` |

#### 2.2.3 词根建设流程

推荐流程：

```text
1. 初始化基础通用词根
2. 初始化常用同义词/近义词/缩略词关系
3. 扫描业务数据库字段
4. 先用基础通用词根做最大匹配分词
5. 无法识别的部分生成业务域词根候选
6. 结合字段名、样例值、表上下文、同表字段做证据链判断
7. 人工确认业务域词根
8. 生成标准字段
9. 建立物理字段到标准字段映射
```

示例：

```text
交易金额
```

如果已有：

```text
交易 -> jiaoyi
金额 -> jine
```

生成：

```text
jiaoyi_jine
```

如果没有 `jiaoyi`，则：

```text
金额 -> 已有基础通用词根 jine
交易 -> 业务域词根候选 jiaoyi
```

生成候选：

```text
jiaoyi_jine
```

但 `jiaoyi` 需要证据链确认后才能进入 `ACTIVE`。

## 3. 标准字段编码规范

标准字段字段：`om_standard_field.field_code`

必须满足：

1. 必须由已登记词根组成。
2. 每一段都必须是合法词根编码。
3. 多词根之间使用英文下划线 `_` 连接。
4. 必须全小写。
5. 不允许英文单词、英文缩写、原始数据库字段名直接进入标准字段。
6. 词根顺序必须按中文字段含义的自然语序排列。
7. 如果字段只包含一个完整语义，也允许标准字段等于单个词根。
8. 标准字段只能由**规范词根**组成，不能由辅助识别词根组成。

标准字段生成公式：

```text
标准字段 = 词根1 + '_' + 词根2 + '_' + ... + 词根N
```

正确示例：

```text
yonghu_mingcheng
kehu_bianma
shouji_haoma
shenfenzheng_haoma
xiangmu_mingcheng
hetong_bianhao
chuangjian_shijian
gengxin_shijian
```

错误示例：

```text
user_name
customer_code
phone_num
id_card
create_time
update_time
```

对应修正：

| 错误字段 | 正确标准字段 |
|---|---|
| `user_name` | `yonghu_mingcheng` |
| `customer_code` | `kehu_bianma` |
| `phone_num` | `dianhua_haoma` 或 `shouji_haoma` |
| `id_card` | `shenfenzheng_haoma` |
| `create_time` | `chuangjian_shijian` |
| `update_time` | `gengxin_shijian` |

### 3.1 标准字段选择标准

生成标准字段时按以下规则选择：

1. 先确认字段中文含义，再拆词根，不从英文名机械翻译。
2. 字段含义能由已有词根表达时，必须复用已有词根。
3. 字段含义无法由已有词根表达时，先新增词根，再新增标准字段。
4. 同义、近义、缩略只影响候选推荐，不改变标准字段必须由合法词根组成的规则。
5. 标准字段应表达字段口径，不表达来源表名、来源系统名、临时加工过程。

示例：

| 中文字段含义 | 推荐标准字段 | 不推荐 |
|---|---|---|
| 项目名称 | `xiangmu_mingcheng` | `project_name` |
| 凭证日期 | `pingzheng_riqi` | `voucher_date` |
| 更新时间 | `gengxin_shijian` | `update_time` |
| 金额 | `jine` | `amount` |

## 4. 词根与标准字段关系

表：`om_root_field_map`

一个标准字段由几个词根组成，就必须建立几条映射。

示例：标准字段 `yonghu_mingcheng`

| 词根 | 标准字段 | 关系类型 |
|---|---|---|
| `yonghu` | `yonghu_mingcheng` | `SYNONYM` |
| `mingcheng` | `yonghu_mingcheng` | `SYNONYM` |

默认规则：

| 字段 | 默认值 |
|---|---|
| `relation_type` | `SYNONYM` |
| `weight_score` | `1.00` |
| `status` | `ACTIVE` |

### 4.1 om_root_field_map 关系类型解释

当前表结构中的 `relation_type` 取值为 `SYNONYM`、`NEAR`、`ALIAS`。在本表中，它不是“词根之间的同义关系”，而是“词根与标准字段之间的关联强度/来源关系”。

| relation_type | 使用条件 | 示例 |
|---|---|---|
| `SYNONYM` | 词根是标准字段的直接组成部分，语义完全命中 | `yonghu` -> `yonghu_mingcheng` |
| `NEAR` | 词根不是字段直接组成部分，但可辅助识别近义字段 | `dianhua` -> `shouji_haoma` |
| `ALIAS` | 词根来自历史叫法、系统别名、业务别名，用于兼容识别 | `gongsi` -> `faren_zhuti` |

强制要求：

1. 标准字段由哪些词根组成，就必须为这些词根写入 `SYNONYM` 映射。
2. `NEAR`、`ALIAS` 只能作为辅助映射，不能替代标准字段自身的组成词根映射。
3. 每个标准字段至少要有一条 `SYNONYM` 映射。
4. 一个标准字段的所有组成词根都必须在 `om_term_root` 中存在。
5. 不允许在 `om_root_field_map` 中表达词根到词根的关系；这类关系必须写入 `om_root_synonym`。

示例：`shouji_haoma`

| root_code | field_code | relation_type | 说明 |
|---|---|---|---|
| `shouji` | `shouji_haoma` | `SYNONYM` | 标准字段直接组成词根 |
| `haoma` | `shouji_haoma` | `SYNONYM` | 标准字段直接组成词根 |
| `dianhua` | `shouji_haoma` | `NEAR` | 电话可辅助识别手机号码，但不是该字段直接组成词根 |

## 5. 同义词根规范

表：`om_root_synonym`

用于维护词根之间的同义、近义、缩略关系。

示例：

| 主词根 | 关联词根 | 关系类型 | 说明 |
|---|---|---|---|
| `shouji` | `dianhua` | `NEAR` | 手机与电话近义，但口径不完全相同 |
| `bianhao` | `bianma` | `NEAR` | 编号与编码近义 |
| `zuzhi` | `bumen` | `NEAR` | 组织与部门近义 |
| `shenfenzheng` | `zhengjian` | `NEAR` | 身份证与证件近义 |

要求：

1. 同义词两端都必须是合法词根。
2. 不允许把标准字段写入同义词表。
3. 同义关系用于辅助建模，不自动代表字段口径完全一致。
4. 关系类型可取：`SYNONYM`、`NEAR`、`ABBREVIATION`。

### 5.1 主词根与关联词根的方向

`om_root_synonym.root_id` 表示**主词根**，也叫推荐词根、规范词根。

`om_root_synonym.synonym_root_id` 表示**关联词根**，也叫同义词根、近义词根、缩略词根。

存储方向统一为：

```text
主词根 root_id -> 关联词根 synonym_root_id
```

含义是：当系统识别到关联词根时，可以归并、提示或推荐到主词根。

示例：

| 主词根 | 关联词根 | 关系类型 | 解释 |
|---|---|---|---|
| `shouji` | `dianhua` | `NEAR` | 手机是更精确的规范词，电话是近义词 |
| `bianhao` | `bianma` | `NEAR` | 若业务字段口径为编号，则编号作为主词根 |
| `shenfenzheng` | `sfz` | `ABBREVIATION` | `sfz` 若经评审保留为缩略词根，只能作为缩略关联 |

注意：

1. 物理存储按单向记录保存，不要求同时插入反向记录。
2. 查询和建模时可以按需要做双向扩展，但标准化落库时必须以主词根为准。
3. 如果两个词根在业务上完全等价，也必须选出一个主词根，不能互相都当主词根。

### 5.2 主词根选择规则

主词根必须按以下顺序选择：

1. 语义更精确者优先。
2. 本项目/本行业标准术语优先。
3. 出现在标准字段中的词根优先。
4. 中文全称优先于简称。
5. 稳定业务概念优先于系统内部叫法。
6. 高频通用词优先于低频局部词。

示例：

| 候选关系 | 推荐主词根 | 原因 |
|---|---|---|
| 手机 / 电话 | `shouji` | 手机号码字段比电话更精确 |
| 编号 / 编码 | 视字段口径选择 | 编号偏业务标识，编码偏分类/代码 |
| 身份证 / 证件 | `shenfenzheng` | 身份证比证件更具体 |
| 组织 / 部门 | `zuzhi` | 组织覆盖范围更通用，部门可作为近义 |

### 5.3 relation_type 选择规则

| relation_type | 使用条件 | 是否可自动等价替换 |
|---|---|---|
| `SYNONYM` | 两个词根在当前业务域内含义等价，替换后不改变字段口径 | 可以作为强推荐，但仍需保留审计 |
| `NEAR` | 两个词根相近但口径不完全一致 | 不可自动替换，只能候选提示 |
| `ABBREVIATION` | 一个词根是另一个词根的简称、缩写或惯用短称 | 可推荐到主词根，但首次需人工确认 |

禁止事项：

1. 不允许把 `NEAR` 当作 `SYNONYM` 使用。
2. 不允许因字段名相似就建立同义关系，必须看业务口径。
3. 不允许把标准字段拆不清时强行写入同义词表。
4. 不允许将英文缩写直接作为词根，除非经评审确认为业务通用缩略词根。
5. 不允许一个关联词根同时指向多个主词根且关系类型为 `SYNONYM`；确有歧义时只能使用 `NEAR` 并交由人工确认。
6. `ABBREVIATION` 的关联词根只能是辅助识别词根，不得参与标准字段组成。

### 5.4 与标准字段的关系边界

`om_root_synonym` 不改变标准字段编码。

例如：

```text
主词根：shouji
近义词根：dianhua
标准字段：shouji_haoma
```

即使存在 `shouji -> dianhua` 的近义关系，标准字段仍然是 `shouji_haoma`，不能自动变成 `dianhua_haoma`。

同义词根只用于：

1. 源字段识别。
2. 建模候选推荐。
3. 问数术语归一。
4. 人工确认时提供候选解释。

不得用于：

1. 自动重写已生效标准字段。
2. 自动认定两个标准字段口径完全一致。
3. 自动合并对象、属性或关系。

## 6. 从源库字段抽象的流程

从达梦或其他源库字段生成词典时，必须按以下流程处理：

1. 读取 schema、table、column、comment。
2. 判断字段真实中文业务含义。
3. 将中文业务含义拆成最小中文业务词。
4. 将每个中文业务词转换为汉语拼音词根。
5. 检查词根是否已存在，不存在则新增。
6. 将词根用下划线组合成标准字段。
7. 写入标准字段。
8. 写入词根与标准字段映射。
9. 必要时写入词根同义关系。

## 6.1 工程化生成决策树

扫描全库字段时，系统不得直接把源字段名机械转换为词根或标准字段，必须先判定信息充分性。

```text
源字段 + 字段注释 + 表注释 + schema/table上下文
        |
        v
是否属于业务范围内的活跃业务表？
        |
        +-- 否 --> 排除或进入待治理，不生成正式词典
        |
        +-- 是 --> 是否存在明确中文注释？
                  |
                  +-- 是 --> 能否按词根词典拆成明确中文业务词？
                  |             |
                  |             +-- 是 --> 生成规范词根 + 标准字段，可自动入库或高置信候选
                  |             |
                  |             +-- 否 --> 进入人工确认，不自动入库
                  |
                  +-- 否 --> 是否命中已维护缩写/别名词典？
                                |
                                +-- 是 --> 生成候选，需带来源说明；高风险字段需人工确认
                                |
                                +-- 否 --> 不生成正式词根和标准字段，只记录待治理项
```

强制前置条件：

1. 只有进入一期业务范围的 schema/table 才参与正式词典生成。
2. 平台配置表、日志表、审批流配置表、报表模板表、临时表、备份表、系统元数据表默认不参与正式词典生成。
3. 全库扫描结果必须先做“表级过滤”，再做字段级抽象。
4. 未通过表级过滤的数据，最多进入待治理清单，不得写入 `ACTIVE` 词典。

表级过滤建议：

| 表类型 | 识别特征 | 默认处理 |
|---|---|---|
| 业务主表/事实表 | 有业务中文表注释，字段为业务对象属性 | 参与生成 |
| 字典/参数表 | 表注释含字典、参数、枚举、配置 | 候选，需确认 |
| 审批/流程/模板表 | 表名或注释含 approval、flow、template、模板、流程 | 默认排除 |
| 日志/审计表 | 表名或注释含 log、audit、日志、记录 | 默认排除，除非业务明确需要 |
| 临时/备份表 | 表名含 temp、tmp、bak、backup、历史备份 | 排除 |
| 平台元数据表 | 表注释含 表定义、字段定义、目标表、源表、模板 | 排除 |

## 6.2 自动生成等级

| 等级 | 条件 | 系统动作 | 是否写入四表 |
|---|---|---|---|
| L1 自动入库 | 字段中文注释明确，能稳定拆词，词根全部合法 | 生成词根、标准字段、映射 | 可以 |
| L2 高置信候选 | 中文注释明确，但拆词存在轻微歧义 | 生成候选，等待人工确认 | 不直接写正式数据 |
| L3 低置信候选 | 只有英文/拼音缩写，但命中已维护别名或缩略词典 | 生成候选与解释 | 不直接写正式数据 |
| L4 待治理 | 无中文注释、未命中缩写词典、无法判断业务含义 | 记录来源字段 | 不生成 |
| L5 禁止生成 | 临时字段、技术字段、日志字段、无法稳定复用字段 | 标记排除 | 不生成 |

L1 可以自动写入；L2/L3 必须人工确认后才能写入；L4/L5 只能进入治理清单。

L1 额外门槛：

1. 表必须通过业务表范围过滤。
2. 字段中文注释不能只是泛化词，例如“名称”“编号”“日期”“状态”“备注”。
3. 中文注释必须能按词根词典或分词规则拆成两个及以上稳定词根；单词根字段只有在该词本身就是完整业务口径时才可 L1。
4. 拆词结果不得产生长串复合词根，例如 `biaobiaoshi`、`mubiaobiaodingyi`、`zhidingjiedianjiaqianjilubiao`。
5. 拆不清的中文注释一律降级为 L2 或 L4。

## 6.3 不知道怎么生成的场景

以下场景系统必须承认“不知道”，不得强行生成：

| 源字段 | 问题 | 处理 |
|---|---|---|
| `KHBH` | 可能是客户编号、开户编号、考核编号 | 待确认 |
| `XM` | 可能是姓名、项目、项目名 | 待确认 |
| `RQ` | 只有“日期”泛义，不知道业务口径 | 可生成 `riqi` 候选，但不得绑定具体业务字段 |
| `BZ` | 可能是备注、币种、标志、步骤 | 待确认 |
| `TYPE` | 英文字段，业务含义不明 | 待确认 |
| `FLAG` | 技术标志，不知道业务含义 | 待治理或排除 |
| `COL1` | 无业务含义 | 禁止生成 |
| `表标识` | 平台元数据口径，不是业务字段 | 表级过滤或 L5 |
| `目标表定义` | 元数据/模板配置口径 | 表级过滤或 L5 |

## 6.4 容易生成错误的场景

以下场景即使能猜到词根，也不得直接自动入库：

| 场景 | 错误风险 | 正确处理 |
|---|---|---|
| 同一缩写多义，如 `BH` | 编号/编码/变化/驳回 | 结合注释和表上下文；无注释则人工确认 |
| 英文名与中文注释冲突 | 英文字段可能历史遗留 | 以中文注释和业务确认优先 |
| 表名带业务域但字段泛化，如 `FIN_PROJECT.NAME` | `NAME` 不一定是项目名称 | 结合表注释和主键关系确认 |
| `ID`、`CODE`、`NO` 类字段 | 不能知道主体是谁 | 必须补主体词根，如 `kehu_bianhao` |
| `STATUS`、`STATE` | 状态口径不明 | 可生成 `zhuangtai` 候选，需对象上下文 |
| `DATE`、`TIME` | 日期/时间类型泛化 | 需确认是创建、更新、业务、发生、结束等哪类时间 |
| 中文短语被整体转拼音 | 会生成不可复用长词根 | 必须先分词，不能整句转拼音 |

## 6.6 中文分词与词根生成规则

中文注释不能直接整句转拼音。必须先拆成中文词，再把每个中文词转成词根。

错误：

```text
表标识 -> biaobiaoshi
目标表定义 -> mubiaobiaodingyi
指定节点加签记录表名称 -> zhidingjiedianjiaqianjilubiao_mingcheng
用户类型 -> yonghuleixing
项目名称 -> xiangmumingcheng
```

正确处理：

| 中文注释 | 处理 |
|---|---|
| 表标识 | 平台元数据字段，默认排除；若业务需要则拆为 `biao` + `biaoshi` |
| 目标表定义 | 平台元数据字段，默认排除 |
| 指定节点加签记录表名称 | 流程配置/元数据表字段，默认排除，不生成业务标准字段 |
| 用户类型 | `yonghu` + `leixing` -> `yonghu_leixing` |
| 项目名称 | `xiangmu` + `mingcheng` -> `xiangmu_mingcheng` |
| 创建时间 | `chuangjian` + `shijian` |
| 客户名称 | `kehu` + `mingcheng` |

分词规则：

1. 优先使用已确认词根中文名进行最长匹配。
2. 优先识别业务对象词，例如客户、项目、合同、员工、组织、凭证。
3. 再识别属性词，例如名称、编号、编码、金额、日期、时间、状态。
4. 最后识别修饰词，例如创建、更新、开始、结束、是否。
5. 未命中词根词典的长中文短语不得自动整体生成词根，必须进入候选。

强制校验：

1. 禁止把明确可拆的中文词组整体转成一个词根。
2. 禁止生成 `yonghuleixing`、`xiangmumingcheng`、`kehuxingming` 这类由多个基础词硬拼出来的词根。
3. 标准字段编码中，凡是已能被权威术语或基础词根拆解的，必须用下划线连接对应词根。
4. 如果系统无法证明 `用户类型` 已拆为 `用户` + `类型`，则不得写入 `ACTIVE`。
5. 每个 `om_standard_field.field_code` 拆分后的每一段，必须在 `om_term_root.root_code` 中存在。

### 6.6.1 不采用“不可拆白名单”

本项目不采用“中文名长度 > 2 且无下划线必须进入不可拆白名单”的方案。

原因：

1. 白名单会变成长期人工维护负担。
2. 新业务域、新系统、新简称会不断出现，白名单无法稳定覆盖。
3. 白名单容易把临时判断固化为标准，后续纠偏成本高。
4. 成熟数据治理实践更常用“业务术语表/权威词条 + 同义词 + 分类 + 审批”治理术语，而不是维护不可拆白名单。

替代方案：

```text
权威业务术语表 + 基础词根词典 + 分词证据链 + 候选审批
```

### 6.6.2 权威业务术语与基础词根

中文词进入词根体系前，先判断它属于哪一类：

| 类型 | 定义 | 示例 | 标准字段中如何使用 |
|---|---|---|---|
| 基础词根 | 最小可复用业务词 | 用户、类型、项目、名称、金额 | 直接作为词根段 |
| 权威业务术语 | 行业或组织内稳定使用的整体概念 | 身份证、营业执照、统一社会信用 | 可作为一个规范词根 |
| 候选术语 | 可能是整体术语，但证据不足 | 风险暴露、预算控制、资金计划 | 进入候选，人工确认 |
| 临时/技术短语 | 模板、流程、配置、表定义等 | 表标题、字段定义、审批节点 | 默认不进入业务词典 |

权威业务术语不是白名单，它是正式业务术语资产，要求：

1. 有中文术语名。
2. 有业务定义。
3. 有所属业务域。
4. 有同义词/近义词关系。
5. 有来源证据。
6. 有状态：候选、已确认、停用。

在当前四表结构下，已确认的权威业务术语可存入 `om_term_root`，例如：

| 术语 | root_code | 说明 |
|---|---|---|
| 身份证 | `shenfenzheng` | 已确认整体业务术语 |
| 营业执照 | `yingyezhizhao` | 已确认整体业务术语 |
| 统一社会信用 | `tongyishehuixinyong` | 已确认整体业务术语 |

但类似 `用户类型`、`项目名称`、`客户编号` 不属于权威业务术语，应拆为基础词根：

| 中文字段 | 正确标准字段 |
|---|---|
| 用户类型 | `yonghu_leixing` |
| 项目名称 | `xiangmu_mingcheng` |
| 客户编号 | `kehu_bianhao` |

### 6.6.3 分词决策规则

系统分词时按以下顺序决策：

1. 先匹配已确认权威业务术语。
2. 再匹配基础词根。
3. 再结合字段证据链判断是否为候选术语。
4. 无法稳定判断时，不生成 `ACTIVE`，进入候选。

示例：

| 中文输入 | 决策 | 结果 |
|---|---|---|
| 身份证号码 | 身份证是权威业务术语，号码是基础词根 | `shenfenzheng_haoma` |
| 用户类型 | 用户、类型都是基础词根 | `yonghu_leixing` |
| 统一社会信用代码 | 统一社会信用是权威业务术语，代码是基础词根 | `tongyishehuixinyong_daima` |
| 表标题 | 命中技术/元数据短语 | 默认排除 |
| 风险暴露金额 | 风险、金额可识别，暴露需候选确认 | 候选，不直接 `ACTIVE` |

### 6.6.4 属性后缀强制拆分规则

工程实现必须内置属性后缀拆分规则，而不能依赖人工逐个维护。

属性后缀表：

| 中文后缀 | 词根 |
|---|---|
| 时间 | `shijian` |
| 日期 | `riqi` |
| 名称 | `mingcheng` |
| 编号 | `bianhao` |
| 编码 | `bianma` |
| 类型 | `leixing` |
| 状态 | `zhuangtai` |
| 金额 | `jine` |
| 号码 | `haoma` |
| 代码 | `daima` |
| 标志 | `biaozhi` |
| 标识 | `biaoshi` |
| 标题 | `biaoti` |
| 说明 | `shuoming` |
| 地址 | `dizhi` |

拆分算法：

```text
如果中文字段名以属性后缀结尾，且剩余前缀非空：
  前缀 = 主体词或动作词
  后缀 = 属性词
  标准字段 = 前缀词根 + '_' + 后缀词根
  词根表只写入前缀词根和后缀词根，不写入整体词根
```

示例：

| 中文字段名 | 错误 | 正确 |
|---|---|---|
| 修改时间 | `xiugaishijian` | `xiugai_shijian` |
| 操作用户 | `caozuoyonghu` | `caozuo_yonghu` |
| 接收用户 | `jieshouyonghu` | `jieshou_yonghu` |
| 父级标识 | `fujibiaoshi` | `fuji_biaoshi` |
| 模型标识 | `moxingbiaoshi` | `moxing_biaoshi` |
| 业务日期 | `yewuriqi` | `yewu_riqi` |
| 交易金额 | `jiaoyijine` | `jiaoyi_jine` |

这条规则优先级高于“整体术语匹配”。也就是说，除非该词被明确确认为权威业务术语，否则属性后缀必须拆分。

建议维护一份基础中文词根表：

| 类型 | 示例 |
|---|---|
| 对象词 | 客户、项目、合同、员工、组织、部门、岗位、凭证、科目 |
| 属性词 | 名称、编号、编码、号码、金额、日期、时间、状态、类型、地址 |
| 动作/修饰词 | 创建、更新、开始、结束、生效、失效、审核、确认 |
| 技术/排除词 | 表、字段、模板、配置、日志、流程、节点、权限、版本 |

命中技术/排除词时，不直接生成业务标准字段，除非该表已被人工确认为业务对象。

### 6.6.1 技术词根与业务词根的双重身份

部分中文词既可能是合法词根，也可能是技术/元数据词。

例如：

| 词 | 可作为词根 | 默认是否生成正式业务字段 |
|---|---|---|
| 表 | 可以 | 否 |
| 字段 | 可以 | 否 |
| 模板 | 可以 | 否 |
| 流程 | 可以 | 否 |
| 版本 | 可以 | 否 |
| 类型 | 可以 | 需要业务对象前缀 |
| 状态 | 可以 | 需要业务对象或动作上下文 |

规则：

1. `表`、`字段`、`模板`、`流程`、`配置`、`版本` 等词可以作为词根存在，但默认只用于平台元数据或候选解释，不进入业务正式标准字段。
2. `表标题`、`表类型`、`字段名称`、`模板编码` 这类字段默认排除。
3. 如果业务系统确实存在“表”作为业务对象，例如餐饮桌台、报表登记对象，必须人工确认业务含义后才能入库。
4. `类型`、`状态`、`标志` 不能单独作为高置信标准字段，必须结合对象上下文生成，如 `kehu_leixing`、`xiangmu_zhuangtai`。

## 6.5 字段上下文推断规则

当字段名本身信息不足时，可以使用上下文辅助推断，但上下文推断不能直接等同于事实。

允许使用的上下文：

1. 字段中文注释。
2. 表中文注释。
3. schema 名称和业务域。
4. 主键、外键、索引名称。
5. 同表其他字段的命名模式。
6. 已确认的词根、标准字段、同义词根。

上下文推断分级：

| 推断来源 | 可信等级 | 是否可自动入库 |
|---|---|---|
| 字段中文注释明确 | 高 | 可以 |
| 表中文注释 + 字段为通用字段，如 `NAME` | 中 | 候选 |
| schema 业务域 + 英文缩写 | 低 | 候选或待确认 |
| 仅字段名缩写 | 很低 | 不入库 |

示例：

| 表注释 | 字段 | 字段注释 | 推荐结果 | 等级 |
|---|---|---|---|---|
| 客户信息表 | `NAME` | 客户名称 | `kehu_mingcheng` | L1 |
| 客户信息表 | `NAME` | 空 | `kehu_mingcheng` 候选 | L2 |
| 无 | `KHBH` | 空 | 待确认 | L4 |

## 6.5.1 字段语义证据链

字段无中文注释或只有缩写时，允许读取少量样例值辅助判断，但样例值只能作为证据之一，不能单独决定业务口径。

字段语义必须按以下证据链综合判断：

```text
字段语义 = 字段名证据 + 样例值证据 + 表上下文证据 + 同表字段证据
```

四类证据定义：

| 证据 | 作用 | 示例 |
|---|---|---|
| 字段名证据 | 提供候选方向 | `KHBH` 可能是客户编号 |
| 样例值证据 | 验证数据形态和值模式 | `KH001` 像编号，`410xxx` 像身份证号 |
| 表上下文证据 | 判断字段所属业务对象 | `CUSTOMER_INFO`、客户信息表指向客户对象 |
| 同表字段证据 | 与同表其他字段互相印证 | `KHMC`、`LXDH`、`DZ` 支持客户资料语境 |

证据链示例：

```text
表：CUSTOMER_INFO
字段：KHBH
样例值：KH001、KH002
同表字段：KHMC、LXDH、DZ

推断：
KHBH = 客户编号
词根 = kehu + bianhao
标准字段 = kehu_bianhao

证据：
字段名 KHBH -> KH 可能是客户，BH 可能是编号
样例值 KH001/KH002 -> 编号形态
表 CUSTOMER_INFO -> 客户对象
同表 KHMC/LXDH/DZ -> 客户名称、联系电话、地址
```

反例：

```text
字段：RQ
样例值：2026-05-20、2026-05-21

只能判断：
该字段是日期形态。

不能判断：
它是创建日期、业务日期、付款日期、发生日期、入职日期。
```

因此 `RQ` 不能仅凭样例值直接生成 `chuangjian_riqi`、`fukuan_riqi` 等具体标准字段，只能作为候选或待确认。

## 6.5.2 证据完整度与处理方式

| 证据完整度 | 处理方式 |
|---|---|
| 4 类证据一致 | 高置信候选；如命中已确认权威业务术语或基础词根规则，可准自动入库 |
| 3 类证据一致 | 候选，必须人工确认 |
| 2 类证据一致 | 低置信候选 |
| 只有 1 类证据 | 不生成正式字段 |
| 证据冲突 | 不生成，进入待治理 |

强制要求：

1. 样例值默认读取 2~5 条即可，不做全表扫描。
2. 样例值不得包含敏感明文落入审计日志；身份证号、手机号等必须脱敏记录。
3. 样例值只能提升候选置信度，不能绕过人工确认。
4. 字段名、样例值、表上下文、同表字段结论冲突时，以“待治理”为准。
5. 生成候选时必须保存证据说明，至少包括字段名、样例值模式、表上下文、同表字段依据。

一句话规则：

```text
字段名负责提出猜测；
样例值负责验证形态；
表上下文负责定位对象；
同表字段负责互相证明。
```

## 6.5.3 语义聚类与覆盖率口径

全库扫描时，不应把每个物理字段都当成一个独立标准字段。大量物理字段可能表达同一个业务语义，必须先做语义聚类，再生成标准字段。

错误口径：

```text
覆盖率 = 已生成标准字段数 / 物理字段总数
```

正确口径：

```text
覆盖率 = 已映射到标准字段的有效业务物理字段数 / 有效业务物理字段总数
```

说明：

1. 28 万物理字段不等于 28 万业务语义字段。
2. 同一业务语义可能在多个表、多个系统、多个历史版本中重复出现。
3. 标准字段应该收敛，而不是膨胀。
4. 一个标准字段可以覆盖多个物理字段。
5. 一期目标中的“覆盖 70%”应理解为**有效业务物理字段映射覆盖率 >= 70%**。

示例：

```text
KHMC
CUSTOMER_NAME
CUST_NAME
客户名称
往来单位名称
```

这些字段可聚类到：

```text
kehu_mingcheng
```

再例如：

```text
RQ
BIZ_DATE
VOUCHER_DATE
发生日期
凭证日期
```

不能简单全部归为 `riqi`，必须结合表上下文、样例值、同表字段区分为：

```text
yewu_riqi
pingzheng_riqi
fasheng_riqi
```

## 6.5.4 语义聚类流程

全库字段标准化应按以下流程执行：

```text
物理字段
  -> 表级过滤
  -> 技术字段过滤
  -> 字段证据链提取
  -> 值模式分析
  -> 语义聚类
  -> 标准字段生成
  -> 词根拆解
  -> 物理字段映射到标准字段
```

语义聚类依据：

| 聚类依据 | 说明 |
|---|---|
| 字段名相似 | 如 `KHMC`、`CUST_NAME`、`CUSTOMER_NAME` |
| 中文注释相似 | 如“客户名称”“往来单位名称” |
| 样例值形态相似 | 如姓名、手机号、身份证号、金额、日期 |
| 表上下文一致 | 如客户表、员工表、凭证表 |
| 同表字段组合一致 | 如 `KHMC/KHBH/LXDH` 指向客户语境 |
| 业务域一致 | 如 finance、investment、hr |

聚类后生成规则：

1. 每个语义聚类只生成一个推荐标准字段。
2. 标准字段必须能拆回规范词根。
3. 每个物理字段需保留映射证据。
4. 同一个物理字段只能有一个主标准字段映射。
5. 可存在候选标准字段，但正式映射必须唯一。

## 6.5.5 标准字段覆盖映射要求

虽然当前四张词典表只保存词根、标准字段和词根关系，但工程实现必须保留“物理字段 -> 标准字段”的映射结果。

映射结果建议进入建模候选或后续映射规则表，例如：

```text
source_schema
source_table
source_column
source_comment
sample_pattern
standard_field_code
confidence_score
evidence_json
status
```

覆盖率统计必须基于该映射结果。

覆盖率计算：

```text
有效业务字段数 = 全库字段数 - 系统字段 - 技术字段 - 临时/备份/模板/日志字段

映射覆盖率 = 已确认或高置信映射字段数 / 有效业务字段数
```

70% 覆盖目标要求：

1. 不要求 70% 字段都生成新的标准字段。
2. 要求 70% 有效业务字段能映射到已有或新生成标准字段。
3. 低置信候选不得计入已覆盖。
4. 仅字段名相似、无其他证据支撑，不得计入已覆盖。
5. 样例值证据、表上下文证据、同表字段证据至少满足两类，才可计入高置信覆盖。

示例：

```text
源字段：USER_NAME
字段注释：用户名称

中文语义：用户名称
拆解词根：用户 + 名称
词根编码：yonghu + mingcheng
标准字段：yonghu_mingcheng
```

## 7. 入库字段要求

### 7.1 om_term_root

| 字段 | 要求 |
|---|---|
| `tenant_id` | 默认租户可用 `1` |
| `root_code` | 汉语拼音，全小写，单一词根 |
| `root_name` | 中文词名 |
| `biz_definition` | 中文业务定义 |
| `domain_name` | 业务域，如 `finance`、`investment`、`hr`、`common` |
| `status` | 默认 `ACTIVE` |

说明：

- 正式可用词根使用 `ACTIVE`。
- 候选词根不建议写入正式四表；如果必须暂存，只能使用 `DRAFT`，不得参与标准字段生成。
- 辅助识别词根必须在 `biz_definition` 中写明：`辅助识别词根；主词根=xxx；来源=xxx`。

### 7.2 om_standard_field

| 字段 | 要求 |
|---|---|
| `tenant_id` | 默认租户可用 `1` |
| `field_code` | 汉语拼音词根下划线组合 |
| `field_name` | 中文字段名 |
| `field_type` | 标准类型，如 `string`、`number`、`date`、`datetime`、`longtext` |
| `biz_definition` | 字段业务口径说明 |
| `combo_key` | 可用 `{编号}_{field_code}` |
| `status` | 默认 `ACTIVE` |

说明：

- 正式标准字段使用 `ACTIVE`。
- 候选标准字段不建议写入正式四表；如果必须暂存，只能使用 `DRAFT`。
- `biz_definition` 必须包含中文口径，不能只写来源字段。
- 来源字段可追加在 `biz_definition` 后部，例如：`口径：客户编号；来源：DM.SCHEMA.TABLE.KHBH`。

### 7.3 om_root_field_map

| 字段 | 要求 |
|---|---|
| `root_id` | 词根ID |
| `standard_field_id` | 标准字段ID |
| `relation_type` | 默认 `SYNONYM` |
| `weight_score` | 默认 `1.00` |
| `status` | 默认 `ACTIVE` |

说明：

- `SYNONYM` 映射代表标准字段的直接组成词根，必须完整覆盖。
- `NEAR`、`ALIAS` 映射只允许在人工确认后写入。
- 候选辅助映射不建议写入正式表；如必须暂存，只能用 `INACTIVE` 或另设候选区。

### 7.4 om_root_synonym

| 字段 | 要求 |
|---|---|
| `root_id` | 主词根ID |
| `synonym_root_id` | 同义/近义/缩略词根ID |
| `relation_type` | `SYNONYM`、`NEAR`、`ABBREVIATION` |
| `weight_score` | 默认 `1.00` |
| `status` | 默认 `ACTIVE` |

说明：

- `SYNONYM` 关系必须人工确认或来自强规则。
- `NEAR` 可以作为候选推荐，但不能驱动自动字段合并。
- `ABBREVIATION` 只能连接主词根与辅助识别词根。
- 不建议把候选同义关系写入正式表；如必须暂存，只能用 `INACTIVE`。

## 7.5 现有表结构下的工程落库策略

当前四张表不再单独创建候选表，候选治理通过 `status`、`source_type`、`confidence_level`、`evidence_json`、`review_note` 承载，因此工程实现必须遵守以下策略：

1. **正式可用数据必须是 `ACTIVE`**。
2. 自动扫描得到的 L2/L3/L4 候选结果可以写入四表，但必须使用 `DRAFT` 或 `REJECTED`，不得直接写入 `ACTIVE`。
3. `DRAFT` 标准字段的映射必须为 `INACTIVE`，只能用于候选覆盖、审核和下一轮学习。
4. `REJECTED` 作为负样本保留，生成器不得反复生成。
5. 来源信息应优先写入 `source_type`、`confidence_level`、`evidence_json`、`review_note`，并在 `biz_definition` 中保留简要可读摘要，格式建议：

```text
口径：{中文业务定义}；来源：{schema}.{table}.{column}；生成方式：{AUTO|MANUAL|REVIEWED}
```

6. 程序读取词典用于正式建模时，只能读取 `status = 'ACTIVE'` 的数据；用于候选生成和覆盖审计时，可以读取 `DRAFT` 作为候选上下文。

## 8. 质量校验规则

入库前必须校验：

1. `root_code` 是否为单一汉语拼音词。
2. `field_code` 是否只由合法词根组成。
3. 标准字段中的每个词根是否已存在。
4. 是否存在英文单词或英文缩写混入。
5. 是否存在数字、空格、特殊符号。
6. 是否存在重复词根。
7. 是否存在重复标准字段。
8. 是否存在标准字段没有词根映射。
9. 是否存在词根同义关系指向不存在的词根。
10. `om_root_field_map` 中每个标准字段是否至少有一条 `SYNONYM` 映射。
11. `om_root_field_map` 的 `SYNONYM` 映射是否覆盖标准字段编码的全部词根段。
12. `om_root_synonym` 是否存在自关联，即 `root_id = synonym_root_id`。
13. `om_root_synonym` 是否存在互相重复记录，即 A->B 与 B->A 同时存在且关系类型相同。
14. `om_root_synonym` 是否存在一个关联词根以 `SYNONYM` 指向多个主词根。
15. 辅助识别词根是否被用于标准字段组成。
16. 辅助识别词根是否被写入 `om_root_field_map` 的 `SYNONYM` 映射。
17. `status != 'ACTIVE'` 的词根、标准字段、映射是否被建模任务读取。
18. `biz_definition` 是否缺失中文口径，只保留了来源字段。
19. `om_term_root` 是否存在字段级复合词根，例如 `xiugaishijian`、`caozuoshijian`、`danjubianhao`。
20. `om_term_root.root_name` 是否以属性后缀结尾且长度大于后缀本身；若是，必须检查是否应拆分。
21. `om_standard_field.field_code` 的每一段是否都是基础词根或权威业务术语，不得是完整字段口径。

## 8.1 四表入库顺序

批量导入或程序生成时，必须按以下顺序入库：

1. 写入 `om_term_root`。
2. 写入 `om_standard_field`。
3. 写入 `om_root_field_map`。
4. 写入 `om_root_synonym`。

原因：

- `om_standard_field.field_code` 依赖词根规范。
- `om_root_field_map` 依赖词根和标准字段都已存在。
- `om_root_synonym` 两端都依赖词根已存在。

## 8.2 自动生成与人工确认边界

从源库字段自动抽象时，系统可以自动生成候选，但必须遵守以下边界：

| 场景 | 是否允许自动入库 | 要求 |
|---|---|---|
| 字段注释明确，能稳定拆出中文词根 | 允许 | 记录来源与生成规则 |
| 字段只有英文缩写，无中文注释 | 不建议直接入库 | 进入待确认队列 |
| 词根同义关系为 `SYNONYM` | 谨慎允许 | 必须有明确业务依据 |
| 词根同义关系为 `NEAR` | 允许候选，不自动替换 | 人工确认后使用 |
| 标准字段无法完整拆回词根 | 不允许 | 必须先补词根 |

工程要求：

1. 自动生成程序必须输出 `confidence_level` 或等价状态：`AUTO`、`CANDIDATE_HIGH`、`CANDIDATE_LOW`、`PENDING_REVIEW`、`REJECTED`。
2. 只有 `AUTO` 可直接写入四表。
3. 其他状态必须进入候选区，不得污染正式词典。
4. 每条自动生成数据必须记录来源字段，至少包含 schema、table、column、comment。
5. 对无中文注释字段，除已维护缩略词典明确命中外，一律不得自动写入正式标准字段。

## 9. 一期边界

一期只要求：

1. 词根、标准字段、映射关系可维护。
2. 当前一期支持 CSV 导出；Excel 导入导出后续再做。
3. 导入失败支持逐行错误明细下载。
4. 词典更新在下次建模任务生效。
5. 已生效语义模型不因词典更新自动变化。

一期不做：

1. 自动把所有英文缩写无人工确认地转成词根。
2. 自动判断复杂字段口径完全等价。
3. 将词根同义关系直接等同于字段口径一致。

## 10. 总结

一句话标准：

```text
词根 = 中文业务词的汉语拼音；
标准字段 = 已登记词根的下划线组合；
任何标准字段都必须能完整拆回词根。
```

## 11. 三轮架构审计结论

### 第一轮：表结构一致性审计

结论：四张表可以支撑词根治理，但必须明确职责边界。

修正点：

1. `om_term_root` 只存单一汉语拼音词根。
2. `om_standard_field` 只存词根组合后的标准字段。
3. `om_root_field_map` 只存词根与标准字段的组成/辅助映射。
4. `om_root_synonym` 只存词根与词根的关系。

### 第二轮：语义边界审计

结论：原规范对 `om_root_synonym` 的方向性、主词根选择和关系类型解释不足。

修正点：

1. `root_id` 是主词根/规范词根。
2. `synonym_root_id` 是关联词根。
3. 存储方向固定为主词根到关联词根。
4. `SYNONYM`、`NEAR`、`ABBREVIATION` 分别对应等价、近义、缩略，不可混用。

### 第三轮：执行与质量审计

结论：自动抽象可以做，但不能越过词根合法性和人工确认边界。

修正点：

1. 标准字段必须能完整拆回已登记词根。
2. `NEAR` 只能用于候选提示，不得自动等价替换。
3. 英文缩写无中文注释时不得直接入库为正式词根。
4. 导入必须按四表依赖顺序执行。

## 13. 最终自洽性结论

本规范在以下前提下自洽：

1. 正式标准字段只由规范词根组成。
2. 辅助识别词根只用于缩略识别，不参与标准字段组成。
3. `om_root_field_map.SYNONYM` 表示词根直接组成标准字段。
4. `om_root_synonym.SYNONYM` 表示词根之间业务等价，二者语义不同，不能混用。
5. 无中文注释、无已确认缩写词典、无可靠上下文时，系统必须输出“待治理”，不能假装自动生成成功。
6. 自动生成只负责提出候选；正式词典必须满足可拆解、可解释、可审计。

工程化一句话：

```text
能确认中文语义才生成；能拆成规范词根才入库；有歧义就候选；无依据就待治理。
```

5. 明确无中文注释字段默认不进入 `ACTIVE` 词典。

```text

只有 L1 且通过质量校验的数据才能进入正式四表。
```
