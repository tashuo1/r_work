> 版本：v1

## 1. 总体开发策略

**策略：M0 基座先行，M1 紧跟并行，M2~M4 加速迭代**

```
第0-1周  M0 轻量单体骨架     ← 单体服务 + 静态产品界面 + API + 验收清单
第1-2周  M1 数据源管理        ← 首个完整闭环（后端→前端→QA→BugDev→验收）
第3-4周  M2 语义建模+词典     ← 最复杂模块，31人天
第5-6周  M3 T+1实例更新      ← ETL引擎 + 调度编排
第7-8周  M4 问数+OpenAPI     ← NL→SQL + 安全闸 + 对外API
第9-10周 M5 稳定性验收       ← 7天观测 + 灰度 + 回滚演练
```

**关键决策：**
- M0 与 M1 部分并行（M0 第1周完成后M1即可启动）
- 每个模块走完整 Agent Team 闭环（Lead分配 → Dev编码 → UX前端 → QA测试+Codex审查 → BugDev修复 → QA回归 → Lead验收）
- 交互留痕到 `logs/YYYY-MM-DD-Mx-模块名.md`

## 2. 团队组成与分工

| 角色 | 代号 | 职责 | 产出 |
|---|---|---|---|
| 架构 Lead | `lead` | 任务分配、方案决策、跨模块协调 | 技术方案、任务卡片 |
| 主力 Dev | `dev` | 核心业务编码、DDL执行、API实现 | Java + Vue 代码 |
| Bug 修复 Dev | `bugdev` | 修bug、review反馈修正、边界处理 | 修复PR + 补测试 |
| QA | `qa` | 用例执行、回归验证、代码审核给出严厉意见 | 测试报告 + 审核意见 |
| UX | `ux` | 前端页面开发、Apple风格交互、可视化 | Vue3 页面 + 组件 |

## 3. Agent Team 执行协议

| 步骤 | 角色 | 动作 | 交付物 | 准入/准出标准 |
|---|---|---|---|---|
| 1 | Lead | 读取WBS任务 → 拆解为具体开发卡片 → 分配给Dev | 任务卡片（含验收用例ID） | 卡片必须绑定WBS编码+用例ID |
| 2 | Dev | 后端编码（Entity/Mapper/Service/Controller/DDL） | Java代码 + 自测记录 | P0用例自测通过 |
| 3 | UX | 前端页面开发（Apple风格） | Vue3页面 + 组件 | 与后端API联调通过 |
| 4 | QA + Codex 并行 | QA：用例执行 + 功能验证 / Codex：代码审查（安全/架构/性能/边界） | QA测试报告 + Codex审查报告 | QA覆盖P0/P1用例；Codex覆盖安全/架构/性能/边界 |
| 5 | BugDev | 修复QA + Codex 发现的所有问题 | 修复PR + 补充测试 | 所有CRITICAL/HIGH问题清零 |
| 6 | QA | 回归验证 | 回归报告 | P0 100%、P1 >=90% |
| 7 | Lead | 最终验收 | 验收结论 | 确认范围未偏离一期边界 |

### Codex 审查维度（4项必审）

1. **安全**：SQL注入、硬编码密钥、权限校验、输入验证
2. **架构**：模块边界、服务内调用链、数据源隔离、DDL合规
3. **性能**：N+1查询、索引使用、连接池配置、缓存策略
4. **边界**：空值处理、并发冲突、超时兜底、幂等保证

## 4. 开发流程

```
Lead 分配任务 → Dev 编码 → UX 前端 → QA 测试 + Codex 审查 → BugDev 修复 → QA 回归 → Lead 验收
```
